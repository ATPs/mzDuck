# mzDuck format design notes

Date: 2026-04-16

## Motivation

### mzPeak format limitations

The mzPeak format (in `../mzPeak/`) stores mass spectrometry data as a ZIP
archive of Parquet files. Each Parquet file uses deeply nested struct columns
(e.g., `point.spectrum_index`, `point.mz`, `point.intensity`). When opened with
DuckDB or pandas, the data appears as a single struct/dict column rather than
flat numeric columns:

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                  point                                  │
│       struct(spectrum_index ubigint, mz double, intensity float)        │
├─────────────────────────────────────────────────────────────────────────┤
│ {'spectrum_index': 0, 'mz': 202.606..., 'intensity': 0.0}              │
│ {'spectrum_index': 0, 'mz': 202.606..., 'intensity': 1938.1174}        │
```

Other issues:

- No built-in indexing across Parquet files; random access relies on Parquet
  page statistics and external JSON indexes.
- The format requires multiple files (data, metadata, chromatograms, index JSON)
  bundled in an uncompressed ZIP, adding complexity.
- The metadata Parquet schema is very complex with nested structs for
  parameters, auxiliary arrays, etc.

### mzsql project (prior art in `../mzsql/`)

The mzsql project (`../mzsql/`) already implements mzML-to-DuckDB conversion
with a simpler schema:

- **MS1**: `filename, id, mz, int, rt`
- **MS2**: `filename, id, premz, fragmz, int, rt`
- **file_info**: `filename, n_scans, rt_start, rt_end`
- **scan_info**: `filename, rt, scan_num, min_mz, max_mz, ms_level, centroided, polarity`

This is good for quick queries but **cannot round-trip back to mzML** because it
is missing:

- precursor charge state
- precursor intensity
- collision energy
- activation type (HCD/CID/ETD)
- isolation window (target, lower, upper offsets)
- base peak m/z and intensity
- total ion current (TIC)
- filter string (Thermo)
- ion injection time
- scan windows
- spectrum native id string (only stores scan number)
- spectrum reference (precursor scan ref)
- run-level metadata (instruments, software, data processing)
- selected ion m/z (distinct from isolation window target)

The mzsql Python converter (`../mzsql/converters/turn_mzml_duckdb.py`) uses
pyteomics and pandas to batch-insert, but accumulates all data in memory as
DataFrames before writing, which won't scale for very large files.

The mzsql R converter (`../mzsql/converters/turn_duckdb_mzml.R`) demonstrates
DuckDB-to-mzML conversion using the Spectra/MsBackendMzR R packages.

### mzDuck approach

mzDuck stores everything in a single DuckDB database file (`.mzduck`).
DuckDB natively supports:

- Flat relational tables with plain numeric columns (DOUBLE, FLOAT, INTEGER).
- B-tree indexes for fast lookup by scan ID, precursor m/z, retention time, etc.
- SQL queries directly on the file without any external tools.
- A single portable file.

Compared to mzsql, mzDuck stores enough metadata for faithful mzML round-trip.
Compared to mzPeak, mzDuck uses a simpler flat schema with built-in indexes.

## Scope

This design covers **MS2-only mzML files** (tandem mass spectrometry data
typical of proteomics DDA/DIA experiments). MS1 spectra, chromatograms, and
wavelength spectra are out of scope for the initial version.

Supported conversions:

- mzML -> mzDuck
- mzDuck -> mzML
- mzDuck -> MGF

## DuckDB schema

### Table: `spectra`

One row per MS2 spectrum. Every column is chosen to live in the mzML source
itself (no mzDuck-assigned surrogate keys) and to use the smallest type that
still round-trips losslessly.

```sql
CREATE TABLE spectra (
    -- identity (only real mzML-observable values)
    scan_number             INTEGER   NOT NULL,   -- parsed from native_id `scan=N` or mzML index
    native_id               VARCHAR,              -- NULL if derivable from native_id_template + scan_number

    -- core scan
    ms_level                UTINYINT  NOT NULL,   -- 1 byte, always 2 in v1
    rt                      FLOAT     NOT NULL,   -- 4 bytes, in run-wide rt_unit

    -- precursor
    precursor_mz            DOUBLE,               -- m/z precision matters for identification
    precursor_charge        TINYINT,              -- 1 byte, signed (negative charges allowed)
    precursor_intensity     FLOAT,
    collision_energy        FLOAT,
    activation_type         VARCHAR,              -- tiny dictionary ('HCD', 'CID', ...)
    isolation_window_target DOUBLE,
    isolation_window_lower  FLOAT,                -- offset, not absolute
    isolation_window_upper  FLOAT,                -- offset, not absolute
    spectrum_ref            VARCHAR,              -- precursor spectrum reference id

    -- spectrum summary
    base_peak_mz            FLOAT,                -- mzML reports float32
    base_peak_intensity     FLOAT,
    tic                     FLOAT,
    lowest_mz               FLOAT,                -- also the pruning hint for XIC
    highest_mz              FLOAT,

    -- per-scan extras
    filter_string           VARCHAR,              -- Thermo filter string (FSST-compresses well)
    ion_injection_time      FLOAT,                -- in run-wide ion_injection_time_unit
    monoisotopic_mz         DOUBLE,               -- Thermo trailer, high precision
    scan_window_lower       FLOAT,
    scan_window_upper       FLOAT,

    -- peak arrays (merged here; no separate spectrum_arrays table)
    mz_array                FLOAT[]   NOT NULL,   -- list element type mirrors source dtype
    intensity_array         FLOAT[]   NOT NULL    -- list element type mirrors source dtype
);
```

Design notes:

- **No `scan_id` and no `source_index`.** Both were mzDuck-invented. `scan_id`
  was a synthetic contiguous identifier and `source_index` mirrored the mzML
  `index` attribute, which is just positional and can be recomputed on export
  from row order. Neither carries information beyond `scan_number`.
- **`scan_number` is the identifier.** It is the integer parsed from
  `native_id` (e.g. `scan=39`) when present, and falls back to the mzML
  `<spectrum index="N">` attribute otherwise. Not declared `PRIMARY KEY` by
  default — see the index section below.
- **`native_id` is stored only when non-templated.** For files with a uniform
  native-id template (every Thermo file, for example: `controllerType=0
  controllerNumber=1 scan=<N>`), the importer stores the template once in
  `run_metadata.native_id_template` and leaves the column NULL. The exporter
  formats the template with `scan_number` at write time. Files with mixed
  templates fall back to storing `native_id` per row.
- **No `selected_ion_*` columns.** The first-draft schema stored both
  `precursor_*` and `selected_ion_*`; in practice these carry the same values
  for v1 (selected ion is preferred and copied into `precursor_*`). The
  activation CV JSON blob is also dropped; `activation_type` plus
  `collision_energy` is sufficient round-trip for v1 activation terms.
- **No `num_peaks` column.** With per-spectrum arrays, `len(mz_array)` gives
  the count in O(1) at query time without storing a redundant integer.
- **Run-constant fields live in `run_metadata`.** `rt_unit`, `polarity`,
  `centroided`, and `ion_injection_time_unit` are uniform across every
  spectrum in the canonical fixture and virtually every real proteomics run.
  They are stored once in `run_metadata`. If the importer observes mixed
  values within a single run it fails with a clear error; promoting them
  back to per-spectrum columns is a schema extension, not a v1 fallback.
- **Type rationale** applied consistently:

  | Column | Type | Why |
  | --- | --- | --- |
  | `scan_number` | INTEGER (INT32) | Up to ~2e9, fits every real instrument. |
  | `ms_level` | UTINYINT | 1 byte. Values 1-10. |
  | `rt` | FLOAT | 4 bytes. ~0.4 ms resolution at 60 min — far below chromatographic scale. |
  | `precursor_charge` | TINYINT | 1 byte. Signed so negative mode is honest. |
  | `precursor_mz`, `isolation_window_target`, `monoisotopic_mz` | DOUBLE | Downstream identification tolerates float32, but m/z precision is the single field users care about most. |
  | everything else numeric | FLOAT | The mzML source is float32 for intensities, base peaks, and injection time. |
  | strings | VARCHAR | DuckDB FSST + dictionary dedups shared prefixes automatically; no benefit to manual encoding. |

- **Peak arrays live on the same row.** The first-draft v1 had a separate
  `spectrum_arrays` table; it is merged into `spectra` here. The relationship
  is strictly 1:1, a join adds nothing, and DuckDB's column-pruned reads make
  metadata-only queries equally cheap either way.

### Peak arrays

Peak arrays live on the `spectra` row as `mz_array FLOAT[]` and
`intensity_array FLOAT[]`. There is no separate `spectrum_arrays` table.

The flat `peaks` shape is still available as a VIEW so user-facing SQL does
not change:

```sql
CREATE VIEW peaks AS
SELECT
    scan_number,
    generate_subscripts(mz_array, 1) - 1 AS peak_index,
    UNNEST(mz_array)        AS mz,
    UNNEST(intensity_array) AS intensity
FROM spectra;
```

Why arrays, not one row per peak (this is the big change vs. the first draft):

The test file's mzML binary arrays are declared as `MS:1000521` 32-bit float
with `MS:1000574` zlib compression. The source peak data is 13.8M × 2 ×
float32 ≈ 110 MB uncompressed and about 55 MB zlib-compressed. mzMLb (HDF5 +
zlib) is 83 MB for the whole file.

Storing peaks as one row per point with `(scan_id BIGINT, peak_index BIGINT,
mz DOUBLE, intensity REAL)` costs:

- 13.8M × 28 bytes of payload = 386 MB raw,
- a `(scan_id, peak_index)` primary-key index on 13.8M rows,
- `idx_peaks_scan_id` on 13.8M rows,
- `idx_peaks_mz` on 13.8M rows of effectively-random doubles (extremely large).

Measured, this produces a ~1.2 GB `.mzduck` file — about 14× mzMLb and 3×
the source mzML. The `idx_peaks_mz` B-tree alone is in the hundreds of MB.

Storing peaks as arrays in one row per spectrum avoids every part of that:

- No redundant `scan_id` stored 13.8M times.
- No explicit `peak_index` column; position in the list is the index.
- No B-tree over every m/z value.
- DuckDB's columnar encoders (ALP / Patas / Chimp for floats, bitpacking /
  dictionary / FSST for ints and strings) see the list's backing storage
  as a plain float column and compress it natively.

Measured on the canonical fixture (55,814 spectra, 13.8M peaks) with `FLOAT[]`
for both arrays and no secondary indexes: `.mzduck` is ~91 MB, on par with
the 83 MB mzMLb. `DOUBLE[]` for m/z pushes it to ~143 MB for no information
gain, because the source is already float32.

Array dtype rules:

- The v1 importer inspects every spectrum's source numpy dtype. If every m/z
  array is `float32`, the column is `FLOAT[]`. If any array is `float64`,
  the column is `DOUBLE[]`. Same rule for intensity.
- The chosen storage dtype is recorded in `run_metadata` under
  `mz_array_storage_dtype` / `intensity_array_storage_dtype` so the mzML
  exporter can round-trip to the correct `MS:1000521` / `MS:1000523`
  CV terms.
- Using `FLOAT[]` is lossless when the source is float32; the exact 32-bit
  bit patterns are preserved because DuckDB `FLOAT` is IEEE 754 binary32,
  the same representation mzML declares with `MS:1000521`.
- Per-array length sanity: m/z and intensity arrays must have equal length.

Per-spectrum pruning and m/z range queries:

- Most useful m/z filtering is done by first narrowing to a small set
  of spectra (by retention time, precursor m/z, or `scan_number`) and
  then unnesting their arrays. The `lowest_mz` / `highest_mz` columns
  on `spectra` let DuckDB skip spectra that cannot contain a target
  fragment:

  ```sql
  SELECT rt, t.mz, t.intensity
  FROM spectra,
       UNNEST(mz_array, intensity_array)
         WITH ORDINALITY AS t(mz, intensity, pos)
  WHERE lowest_mz <= 175.120 AND highest_mz >= 175.118
    AND t.mz BETWEEN 175.118 AND 175.120;
  ```

- A full-peak-universe B-tree index on m/z is intentionally omitted. It was
  the single biggest contributor to file size in the first draft and only
  paid off for the narrow "find every peak in range across every spectrum"
  query, which the spectra-level prune above already satisfies.

### Table: `run_metadata`

Key-value store for run-level metadata needed for mzML round-trip, plus the
run-constant fields hoisted out of `spectra`.

```sql
CREATE TABLE run_metadata (
    key   VARCHAR PRIMARY KEY,
    value VARCHAR                -- JSON-encoded for complex values
);
```

This stores:

| Key | Description |
| --- | --- |
| `schema_version` | mzDuck schema version (e.g. "1") |
| `source_file` | Original mzML filename |
| `run_id` | mzML run id attribute |
| `run_start_time` | mzML run startTimeStamp |
| `instrument_config_ref` | Default instrument configuration ref |
| `source_file_ref` | Default source file ref |
| `rt_unit` | Run-wide RT unit, `minute` or `second` |
| `polarity` | Run-wide polarity, `positive` or `negative` |
| `centroided` | `true` for centroid, `false` for profile |
| `ion_injection_time_unit` | Usually `millisecond` |
| `native_id_template` | e.g. `controllerType=0 controllerNumber=1 scan={scan_number}`; NULL if absent or non-uniform |
| `mz_array_storage_dtype` | `FLOAT` or `DOUBLE`, chosen at import to match source |
| `intensity_array_storage_dtype` | `FLOAT` or `DOUBLE`, chosen at import to match source |
| `compression` | `zstd`, `auto`, or `uncompressed` |
| `compression_level` | Requested zstd level as decimal string |
| `file_description_xml` | Raw XML of `<fileDescription>` |
| `software_list_xml` | Raw XML of `<softwareList>` |
| `instrument_configuration_xml` | Raw XML of `<instrumentConfigurationList>` |
| `data_processing_xml` | Raw XML of `<dataProcessingList>` |
| `referenceable_param_groups_xml` | Raw XML of `<referenceableParamGroupList>` |
| `cv_list_xml` | Raw XML of `<cvList>` |

Storing the header XML sections as raw strings enables exact mzML header
reconstruction without needing to model every possible CV term.

If the importer observes mixed `rt_unit`, `polarity`, `centroided`, or
`ion_injection_time_unit` values across spectra within a single run, it
fails with a clear error rather than silently hoisting the majority value.

### Indexes (opt-in)

**Default: no secondary indexes.** This matches mzMLb, which ships with zero
SQL-style indexes. A freshly-imported `.mzduck` has only DuckDB's built-in
column zonemaps — range queries on `scan_number`, `rt`, or `precursor_mz`
still prune at the storage-block level without any explicit index.

The CLI exposes a single opt-in flag when per-row lookups dominate the
workload:

- `--index-scan-number`: creates `CREATE INDEX idx_spectra_scan_number ON
  spectra(scan_number)` after import and before the final `CHECKPOINT`.
  Default: not set.

Rationale: `scan_number` is the natural primary identifier and the only
column where users routinely do single-row lookups (`WHERE scan_number =
12345`). Every other access pattern — RT windows, precursor m/z ranges,
activation-type filters, product-ion XIC — is a range or full-table scan
over spectrum-level data that DuckDB's zonemaps already handle efficiently
for 55k-scale row counts.

If a specific workflow justifies additional indexes (for example a
retention-time joiner running hundreds of lookups per second), the user
can add them directly at query time:

```sql
CREATE INDEX idx_spectra_rt ON spectra(rt);
CREATE INDEX idx_spectra_precursor_mz ON spectra(precursor_mz);
```

These are not created automatically because each one measurably grows the
`.mzduck` file and most users will never run the query that needs them.

The first-draft v1 schema mandated `idx_peaks_scan_id`, `idx_peaks_mz`,
`idx_spectra_rt`, `idx_spectra_precursor_mz`, `idx_spectra_native_id`, and
`idx_spectra_scan_number`. All are dropped by default:

- `idx_peaks_scan_id` / `idx_peaks_mz` — there is no `peaks` table anymore.
  `idx_peaks_mz` on 13.8M random doubles was the single biggest file-size
  contributor in the first draft.
- `idx_spectra_*` — demoted to either opt-in (`scan_number`) or user-added
  (all others).

### Storage compression

mzDuck forces DuckDB's zstd column compressor by default, applied at the
first CHECKPOINT after bulk import. The CLI exposes this directly:

- `--compression {zstd,auto,uncompressed}`: default `zstd`. Sets DuckDB's
  `force_compression` setting before data is written. `auto` lets DuckDB pick
  per-column (ALP / Patas / Chimp / bitpacking / FSST / zstd).
- `--compression-level N`: default `6`. Hint passed through to zstd when
  DuckDB supports it; silently ignored on builds that don't, with a single
  warning recorded under `run_metadata.conversion_warnings`.

The importer executes `CHECKPOINT` after `create_indexes` so the written
blocks pick up the forced compression instead of staying in WAL pages.

On the canonical fixture, `force_compression=zstd` and `auto` produce
near-identical sizes because the dominant column type is float and DuckDB's
ALP/Patas codecs already match zstd on floats. The `--compression` knob is
mostly valuable for the text metadata columns and as a user-facing handle
for future tuning.

## Activation type mapping

The mzML `<activation>` element uses CV terms. pyteomics exposes them as
dictionary keys:

| pyteomics key                                  | mzDuck `activation_type` |
|------------------------------------------------|--------------------------|
| `beam-type collision-induced dissociation`     | `HCD`                    |
| `collision-induced dissociation`               | `CID`                    |
| `electron transfer dissociation`               | `ETD`                    |
| `electron capture dissociation`                | `ECD`                    |
| `infrared multiphoton dissociation`            | `IRMPD`                  |
| `supplemental collision-induced dissociation`  | `SID`                    |
| other                                          | stored as-is             |

The reverse mapping is needed for mzML export.

## Conversion design

### mzML -> mzDuck

Use `pyteomics.mzml.MzML` to read. Processing flow:

1. Open the mzML file with `pyteomics.mzml.MzML(path)`.
2. Open the DuckDB file and set compression config before writing:
   - `SET force_compression = '<user choice>';` (default `zstd`).
   - Store the compression choice and level in `run_metadata` for provenance.
3. Extract run-level metadata:
   - Parse the mzML XML header using `lxml.etree` to extract the raw XML
     sections (fileDescription, softwareList, etc.).
   - Store each section as a string in `run_metadata`.
4. Two-pass (or promote-on-first-sight) dtype decision for peak arrays:
   - Track observed numpy dtypes for m/z and intensity arrays.
   - Final column types are `FLOAT[]` if every spectrum was float32,
     otherwise `DOUBLE[]` for that column.
   - Record the chosen dtype in `run_metadata.mz_array_storage_dtype` /
     `intensity_array_storage_dtype`.
5. Detect run-constant fields from the first spectrum. For every subsequent
   spectrum, assert that `rt_unit`, `polarity`, `centroided`, and
   `ion_injection_time_unit` are unchanged. Any change is a hard error.
   Store the four values in `run_metadata`.
6. Detect `native_id_template`: parse `scan=<N>` from each `native_id`. If
   the non-numeric prefix (and `scan_number`-as-suffix shape) is uniform,
   store the template in `run_metadata.native_id_template` and skip the
   per-row `native_id` column. Otherwise emit `native_id` per row.
7. For each spectrum, extract:
   - `scan_number` from `native_id`'s `scan=<N>` token; fall back to the
     mzML `<spectrum index="N">` attribute if absent.
   - All other `spectra` columns.
   - m/z and intensity numpy arrays to be bound to `mz_array` /
     `intensity_array` on the same row.
8. Batch-insert for performance:
   - Build batches of `batch_size` spectra (default 5,000) as a single
     pyarrow table that already includes `mz_array` and `intensity_array`
     as `ListArray` columns of the resolved element dtype. A `ListArray`
     is a flat values buffer plus offsets, so per-batch memory is
     proportional to the current batch's peak total, not the file's peak
     total.
   - Register the pyarrow table under a temporary name and `INSERT INTO
     spectra SELECT * FROM ...` from it.
   - Release batch buffers after each insert.
9. Create the `peaks` view.
10. If `--index-scan-number` was supplied, create
    `idx_spectra_scan_number`. No other indexes are created by default.
11. Run `CHECKPOINT` so the forced compression setting is applied to the
    stored blocks, then run the v1 validation queries and close cleanly.

Retention time handling:

- pyteomics returns `scan start time` as a `unitfloat` with `.unit_info`.
- For this test file, unit is "minute" (UO:0000031).
- Store the value as-is in the `rt` column. Record the unit in `run_metadata`.
- If unit is "second", store as-is and record "second".

### mzDuck -> MGF

MGF only stores MS2 spectra. For each spectrum:

```text
BEGIN IONS
TITLE=<native_id>
PEPMASS=<precursor_mz> <precursor_intensity>
CHARGE=<precursor_charge>+
RTINSECONDS=<rt * 60>
<mz1> <intensity1>
<mz2> <intensity2>
...
END IONS
```

Notes:

- RTINSECONDS requires conversion from minutes to seconds if `rt_unit` is
  "minute".
- PEPMASS includes intensity if available (space-separated).
- CHARGE uses the `n+` or `n-` format.
- Peaks are ordered by m/z.
- pyteomics `mgf.write()` can be used, or we can write directly for more
  control.


### mzDuck -> mzML

Use `lxml.etree` to write mzML XML (or `psims.mzml.MzMLWriter` if installed).
Processing flow:

1. Read `run_metadata` to reconstruct the mzML header.
2. Write the raw XML header sections from stored strings.
3. For each spectrum (ordered by `scan_number`):
   - Read the `spectra` row, including `mz_array` and `intensity_array` on
     the same row. The arrays come back as numpy arrays of the stored
     dtype (`float32` or `float64`), matching the original mzML binary
     arrays byte-for-byte.
   - Reconstruct `native_id` from `run_metadata.native_id_template` and
     `scan_number` when `spectra.native_id` is NULL.
   - Build the `<spectrum>` XML element with all metadata CV params,
     pulling `rt_unit`, `polarity`, `centroided`, and
     `ion_injection_time_unit` from `run_metadata`.
   - Encode m/z and intensity arrays as base64 binary data arrays, using
     `MS:1000521` / `MS:1000523` according to the stored dtype recorded in
     `run_metadata`.
4. Write the mzML index at the end.

The resulting mzML should be semantically equivalent to the input, though not
byte-identical (XML formatting, base64 encoding, element ordering may differ).

Alternative approach: `msconvert` can validate our output by re-reading it.

## Useful queries enabled by mzDuck

```sql
-- Find all MS2 spectra for a precursor mass within 10 ppm
SELECT * FROM spectra
WHERE ABS(precursor_mz - 500.25) / 500.25 * 1e6 < 10;

-- Extract ion chromatogram (XIC) for a product ion.
-- Pruning on lowest_mz / highest_mz lets DuckDB skip spectra that cannot
-- possibly contain a peak in the range, so the UNNEST only runs on a small
-- subset even without a peaks-level index.
SELECT rt, SUM(t.intensity) AS xic
FROM spectra,
     UNNEST(mz_array, intensity_array)
       WITH ORDINALITY AS t(mz, intensity, pos)
WHERE lowest_mz <= 175.120 AND highest_mz >= 175.118
  AND t.mz BETWEEN 175.118 AND 175.120
GROUP BY rt ORDER BY rt;

-- Count spectra per charge state
SELECT precursor_charge, COUNT(*) FROM spectra GROUP BY precursor_charge;

-- Get spectrum with highest TIC
SELECT * EXCLUDE (mz_array, intensity_array) FROM spectra
ORDER BY tic DESC LIMIT 1;

-- Find precursors of a specific fragment (like mzsql)
SELECT DISTINCT precursor_mz, precursor_charge, rt
FROM spectra,
     UNNEST(mz_array) AS t(mz)
WHERE lowest_mz <= 175.120 AND highest_mz >= 175.118
  AND t.mz BETWEEN 175.118 AND 175.120;

-- Get all peaks for a specific spectrum (via the peaks view)
SELECT mz, intensity FROM peaks
WHERE scan_number = 42 ORDER BY peak_index;

-- Or directly from the arrays (no UNNEST, faster for single-spectrum reads)
SELECT mz_array, intensity_array
FROM spectra WHERE scan_number = 42;

-- Retention time window extraction
SELECT * EXCLUDE (mz_array, intensity_array) FROM spectra
WHERE rt BETWEEN 10.0 AND 20.0;
```

```python
class MzDuckFile:
    """Read/write access to a .mzduck file."""

        cls,
        mzml_path,
        output_path,
        *,
        overwrite=False,
```

## Implementation plan

### Phase 1: mzML -> mzDuck conversion

1. Implement `schema.py` with the single `spectra` table (metadata plus
   `mz_array` / `intensity_array`), `run_metadata`, and the `peaks`
   view. No mandatory secondary indexes.
2. Implement `mzml_to_mzduck.py`:
   - Read mzML with pyteomics.
   - Extract header XML with lxml for run_metadata.
   - Apply `SET force_compression = '<...>'` on the output connection.
   - Pre-scan the source to confirm uniform `rt_unit`, `polarity`,
     `centroided`, `ion_injection_time_unit`; hoist them to
     `run_metadata`. Detect `native_id_template`.
   - Resolve the common numpy dtype for `mz_array` / `intensity_array`
     and type those columns `FLOAT[]` or `DOUBLE[]` accordingly.
   - Batch-insert one row per spectrum (metadata + both arrays) into
     `spectra` (never one row per peak, no separate arrays table).
   - If `--index-scan-number` is passed, create
     `idx_spectra_scan_number` after insert, then `CHECKPOINT`.
3. Test on `1556259.mzML`:
   - Verify spectrum count matches (55,814).
   - Verify total peak count matches (13,819,715).
   - Spot-check first/last/random spectra against pyteomics direct read.
   - Report file size and conversion time.
   - File-size target: close to `1556259.mzMLb` (~83 MB). Historical
     baseline on the first schema draft was ~1.2 GB.

## Validation strategy

- **Data integrity**: After mzML -> mzDuck, read back every spectrum and compare
  m/z and intensity arrays against pyteomics mzML reader output. Values should
  match: float32 intensity exact, float64 m/z exact (float32 values stored in
  float64 are exact).
- **MGF comparison**: Use msconvert to produce a reference MGF. Parse both MGFs
  and compare spectrum-by-spectrum: precursor m/z, charge, peak count, peak
  values.
- **Round-trip**: mzML -> mzDuck -> mzML -> read with pyteomics. Compare against
  original mzML read. Retention times, precursor info, peak arrays should match.
- **File size**: Compare mzDuck file size against input mzML (409 MB) and mzMLb
  (84 MB). Target is within 10-30% of mzMLb; expected 85-115 MB for the
  canonical fixture. Any file much larger than mzMLb should be treated as a
  regression and investigated before shipping.

## Open questions

1. **Schema versioning**: Store `schema_version = "1"` in `run_metadata` so
   future code can handle older mzDuck files. (Proposed answer: yes, do it.)

2. **Large file performance**: For 55,814 spectra stored as 55,814
   `spectra` rows (not 13.8M peak rows), DuckDB should handle this
   easily. Benchmark insert and query times during Phase 1.

3. **MS1 support (future)**: If we later want MS1 spectra, add them to
   the same `spectra` table with `ms_level = 1` and NULL precursor
   fields. The array storage generalizes cleanly: MS1 scans just have
   longer `mz_array` / `intensity_array` values.

4. **psims dependency**: psims provides a clean mzML writer but is not
   currently installed. For Phase 3, we can either install it or write mzML
   directly with lxml. Decision deferred to Phase 3.

5. **Compression tuning**: `--compression-level` is exposed now even though
   DuckDB 1.5.0 does not yet accept a numeric zstd level via PRAGMA. The
   importer records the requested level in `run_metadata` and writes a
   warning if the current DuckDB cannot honor it, so the CLI stays stable
   across DuckDB upgrades.

6. **Alternative mz dtype storage**: ALP / Patas / Chimp can sometimes beat
   raw float32 for sorted m/z arrays. The v1 importer always stores
   `FLOAT[]` when the source is float32 for exact round-trip; the per-column
   compression choice is left to DuckDB. If future benchmarks show a clear
   win from delta-encoding m/z, it can be added as an opt-in option without
   changing the on-disk column type.
