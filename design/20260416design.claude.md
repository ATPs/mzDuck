# mzDuck V2 Design Notes

Original date: 2026-04-16  
Updated for implementation: 2026-04-17

## One-line summary

mzDuck v2 now stores MS2 as a physical `mgf` payload table plus an optional `ms2_spectra` detail table, keeps exact-or-raw text reconstruction, omits empty optional tables, supports `.mzML.gz`, and can export physical relations as parquet folder or parquet-zip containers.

## Why this changed

The previous draft moved everything into one canonical `ms2_spectra` table and derived `mgf` as a view.

That solved some duplication, but it also removed the practical distinction between:

- full/default storage
- `--ms2-mgf-only`

The user wanted that distinction back.

So the implemented design returns to a physical `mgf` table while still removing duplication from the richer detail layer.

## Implemented split

### Physical `mgf`

Stores:

- scan number
- source order
- RT
- precursor payload
- peak arrays

Does not store:

- `title`

`TITLE` stays derived.

### Physical `ms2_spectra`

Stores only non-MGF metadata:

- instrument configuration ref
- collision energy
- activation type
- isolation window fields
- precursor scan number
- base peak / TIC summary fields
- ion injection time
- monoisotopic m/z
- scan window limits

This table is keyed by `scan_number` and no longer duplicates payload arrays or precursor payload.

## Exact-or-raw rule

The design still follows the same core philosophy:

- if reconstruction is exact for the whole selected run, store the rule
- if not, store the original exact value

That policy applies to:

- `native_id`
- `spectrum_ref`
- `filter_string`

## Filter strings

Filter strings are still run-specific.

- encoder candidates are tested per file
- no candidate is trusted unless it matches every selected row exactly
- if no exact rule exists, raw strings are stored

This keeps the Thermo handling conservative instead of over-generalizing it.

## Empty-table omission

If an optional table ends up empty, it is dropped before final output.

This now applies to both:

- DuckDB `.mzduck`
- parquet / parquet-zip physical exports

## Gzip and parquet

The implementation now supports:

- `.mzML`
- `.mzML.gz`
- `.mzduck`
- parquet folder output
- parquet zip output

The parquet modes are CLI-only. The Python conversion API still returns an opened DuckDB-backed mzDuck handle.

## Practical effect

The resulting storage model is easier to explain:

- `mgf` is the physical MS2 payload contract
- `ms2_spectra` is the physical richer detail layer
- `--ms2-mgf-only` is truly smaller and simpler
- default/full output keeps the detail needed for faithful mzML export
