# mzDuck V2 Design Notes

Date: 2026-04-16  
Updated: 2026-04-17

## One-Sentence Design

mzDuck v2 keeps one canonical MS2 table, derives the public MGF contract as a
view, stores only sparse exact fallbacks for non-reconstructable text, and
always finishes with a compact database copy.

## Why V2 Changed

The earlier layout carried too much duplicated MS2 information:

- MGF-oriented fields were stored separately from detail fields;
- `title` was stored even though it is deterministic;
- `spectrum_summary` duplicated information already available from metadata and
  direct table counts;
- exact precursor references and filter strings were stored row-by-row even
  when they followed simple run-level rules.

The v2 redesign removes those sources of duplication without weakening the
round-trip contract.

## Key Storage Decisions

### Keep one physical MS2 table

`ms2_spectra` is now the real storage layer for MS2.

It holds:

- scan number
- source index
- optional scan-level instrument configuration reference
- precursor typed fields
- scan summary fields
- peak arrays

This table is the only physical MS2 source for new v2 files.

### Make `mgf` a view

The public MGF-shaped relation still exists, but it is derived from
`ms2_spectra`.

This keeps user-facing SQL simple while avoiding duplicate physical storage.

### Replace stored `spectrum_ref`

Instead of storing the full precursor reference string on every row, store:

- `precursor_scan_number` in `ms2_spectra`
- a run-level `spectrum_ref_template` when exact
- sparse overrides only when needed

### Replace stored `title`

MGF `TITLE` is deterministic:

```text
{mgf_title_source}.{scan_number}.{scan_number}.{precursor_charge}
```

There is no need to store it physically in v2.

### Remove `spectrum_summary`

Summary counts belong in `run_metadata`, not in a duplicate per-spectrum table.

Inspection now uses:

- metadata counts and ranges;
- direct table counts;
- direct queries against canonical tables.

## Exact-Or-Raw Philosophy

The core rule is simple:

If reconstruction is exact for the whole selected run, store the rule.
If it is not exact, store the original value.

That policy now drives:

- `native_id`
- `spectrum_ref`
- `filter_string`

### Filter strings

Filter-string handling is deliberately conservative.

- Detection is per run.
- Rules live in a registry.
- A rule must match the entire selected run exactly.
- If no rule matches exactly, mzDuck stores the raw string.

This avoids the old trap of assuming one Thermo-specific reconstruction formula
globally.

## Extra Params

The `spectrum_extra_params` table is for mzML params that are real extras, not
for fields that have a better typed or rule-based representation.

A useful lesson from the PRIDE acceptance run:

- `instrumentConfigurationRef="IC2"` appeared 28,014 times.
- Keeping it as generic extra-param rows inflated the file.
- Promoting it to a typed per-scan field both reduced size and exported more
  faithfully back to mzML.

This is the general v2 principle:

- if a repeated field is structurally important and maps cleanly to mzML, give
  it a typed home;
- reserve `spectrum_extra_params` for the remainder.

## Compaction

The DuckDB write path matters as much as the schema.

Empirically, `VACUUM` did not shrink the PRIDE output enough.
The successful pattern was:

1. write into a staging database;
2. checkpoint;
3. copy into a fresh final database with `COPY FROM DATABASE`;
4. validate;
5. atomically rename.

That is now part of normal v2 output behavior.

## Reader And Export Notes

### Reader expectations

`MzDuckFile.get_spectrum()` should feel like the original spectrum, not like a
half-decoded row.

That means it reconstructs:

- `title`
- `native_id`
- `spectrum_ref`
- `filter_string`

It also promotes scan-level structural fields such as
`instrument_configuration_ref`.

### mzML export expectations

The v2 exporter should not only dump peaks.
It should rebuild a useful mzML document by:

- exporting spectra in source order;
- rebuilding typed fields from canonical storage;
- merging sparse overrides and extra params;
- restoring stored header fragments;
- restoring `defaultDataProcessingRef`;
- restoring per-scan `instrumentConfigurationRef`.

## Measured Outcome On The PRIDE Run

Observed sizes:

| File | Size |
| --- | ---: |
| mzML | 408.44 MiB |
| mzMLb | 83.26 MiB |
| old mzDuck | 120.01 MiB |
| current v2 mzDuck | 90.26 MiB |

Validated exact recovery:

- scan order
- `native_id`
- `spectrumRef`
- Thermo filter strings
- scan-level instrument configuration references

So the current v2 result is not identical to mzMLb size, but it is much closer
than the old mzDuck layout while preserving the important validated fields.

## Practical Interpretation

The v2 architecture should be treated as the default baseline going forward:

- canonical MS2 storage in `ms2_spectra`
- derived public `mgf`
- no physical `spectrum_summary`
- exact-or-raw text reconstruction
- compact-copy finalization

Future storage work should follow the same pattern:

- hoist exact run-level invariants into metadata;
- use typed columns for structurally important repeated fields;
- keep sparse fallbacks for exceptions;
- only use generic extra-param tables when there is no cleaner typed form.
