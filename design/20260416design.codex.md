# mzDuck V1 format and package specification

Date: 2026-04-16

Status: V1 implementation specification

This document is the development guide for the first mzDuck implementation. It
defines the `.mzduck` file contract, the Python package API, the command-line
interface, conversion algorithms, failure behavior, and validation plan. The
intent is that coding can start from this document without re-deciding the
architecture.

## 1. Summary

mzDuck stores centroid MS2 mzML data in a single DuckDB database file. The v1
format is intentionally relational and flat:

- spectrum-level metadata is stored in `spectra`;
- peak arrays are stored in `peaks`;
- run-level and provenance metadata is stored in `run_metadata`;
- common search paths are accelerated with DuckDB indexes.

The v1 Python package is named `mzduck`. It must support:

- mzML -> mzDuck;
- mzDuck -> MGF;
- mzDuck -> mzML;
- direct SQL queries against the `.mzduck` file;
- programmatic spectrum retrieval by `scan_id`.

The v1 mzML writer must use `psims.mzml.MzMLWriter`. Exported mzML files must be
semantically equivalent to the source for supported MS2 spectra, but they are
not expected to be byte-identical to the input XML.

## 2. Goals, non-goals, and scope

### 2.1 Goals

V1 must provide a practical development-quality format and package for MS2
proteomics data:

1. Use one portable `.mzduck` DuckDB file per mzML run.
2. Store enough MS2 spectrum metadata to reconstruct useful mzML and MGF output.
3. Make common MS2 queries simple SQL over flat numeric columns.
4. Avoid the full-run in-memory DataFrame pattern used by early mzsql scripts.
5. Preserve source ordering and source identifiers so exported files remain
   traceable to the original mzML.
6. Make the schema explicit enough for future readers in Python, R, or Rust.

### 2.2 V1 supported input

V1 supports mzML files with centroid MS2 spectra. The canonical test input is:

```text

```

### 2.3 V1 outputs

V1 must write:

- `.mzduck` DuckDB database files;
- `.mgf` files;
- `.mzML` files written through `psims`.

### 2.4 V1 non-goals

V1 does not support:

- MS1 import or export as a required feature;
- chromatograms;
- wavelength spectra;
- profile-mode arrays;
- mzMLb input;
- vendor RAW input;
- multiple mzML files in one `.mzduck`;
- multi-run mzML files;
- byte-identical mzML reconstruction;
- lossless preservation of every arbitrary vendor XML extension.

Unsupported content must not be silently misrepresented. The converter must
either reject unsupported content with a clear error, or skip it only when an
explicit user option requests filtering.

### 2.5 Future extension points

The schema is allowed to evolve to support MS1 later by adding rows to the same
`spectra` and `peaks` tables with `ms_level = 1` and nullable precursor fields.
V1 code must not rely on every future spectrum having precursor data. However,
the v1 converter should fail on unexpected MS1 content unless the user explicitly
chooses an MS2-only import mode.

Chromatograms and wavelength spectra should be added later as separate tables,
not overloaded into `peaks`.

## 3. Background and prior art

### 3.1 mzPeak limitations motivating mzDuck

The sibling mzPeak project stores data as an uncompressed ZIP archive containing
multiple Parquet files plus a JSON index. This design is rich and round-trips
well through the Rust implementation, but it is not ideal for simple SQL use:

- data appears as nested struct/list columns in DuckDB and pandas;
- random access depends on Parquet page statistics and external metadata;
- metadata tables are complex because they model broad mzML/mzdata concepts;
- users need custom readers instead of opening one relational database file.

mzDuck chooses a narrower scope and a flatter relational model.

### 3.2 mzsql prior art

The sibling mzsql project already converts mzML files into DuckDB/SQLite tables:

- `MS1(filename, id, mz, int, rt)`;
- `MS2(filename, id, premz, fragmz, int, rt)`;
- `file_info(filename, n_scans, rt_start, rt_end)`;
- `scan_info(filename, rt, scan_num, min_mz, max_mz, ms_level, centroided, polarity)`.

That model is excellent for quick range queries, but it does not store enough
metadata for meaningful mzML round-trip. Missing fields include charge state,
precursor intensity, activation type, collision energy, isolation window,
base peak, TIC, filter string, injection time, scan windows, native spectrum id,
precursor spectrum reference, and run-level metadata.

mzDuck keeps the queryability of mzsql while storing the MS2 metadata needed for
MGF and mzML export.

## 5. mzDuck file contract

A `.mzduck` file is a DuckDB database file. It must be valid DuckDB and must
contain the tables and indexes defined below.

V1 supports exactly one mzML run per `.mzduck` file. The source mzML basename,
absolute source path when available, and source run id are stored in
`run_metadata`.

### 5.1 Naming and compatibility

- File extension: `.mzduck`.
- Schema version key: `run_metadata['schema_version'] = '1'`.
- Package code must reject newer major schema versions unless it explicitly
  knows how to read them.
- Writers must create all v1 tables, even if optional columns contain only NULL.
- Readers must tolerate additional future columns but must require the v1
  columns listed here.

### 5.2 Type conventions

Use DuckDB SQL types:

| Logical value | DuckDB type | Notes |
|---|---|---|
| IDs and counts | `BIGINT` | Use `BIGINT` for future large files. |
| MS level and charge | `INTEGER` | Charge may be NULL. |
| m/z | `DOUBLE` | Preserve precision for calculations. |
| intensity | `REAL` by default | Promote only if source requires it. |
| retention time | `DOUBLE` | Stored in source unit. |
| text | `VARCHAR` | Use NULL for absent values. |
| booleans | `BOOLEAN` | Use NULL when unknown. |
| timestamps | `VARCHAR` | ISO-8601 string in metadata. |
| structured metadata | `VARCHAR` | JSON string. |

DuckDB does not enforce all foreign keys efficiently for this use case. The
writer must enforce invariants in code and validate them after import.

### 5.3 Table: `spectra`

One row per imported MS2 spectrum.

```sql
CREATE TABLE spectra (
    scan_id                    BIGINT PRIMARY KEY,
    source_index               BIGINT NOT NULL,
    native_id                  VARCHAR NOT NULL,
    scan_number                BIGINT,
    ms_level                   INTEGER NOT NULL,

    rt                         DOUBLE NOT NULL,
    rt_unit                    VARCHAR NOT NULL,

    precursor_mz               DOUBLE,
    precursor_charge           INTEGER,
    precursor_intensity        DOUBLE,
    selected_ion_mz            DOUBLE,
    selected_ion_intensity     DOUBLE,
    selected_ion_charge        INTEGER,
    collision_energy           DOUBLE,
    activation_type            VARCHAR,
    activation_cv              VARCHAR,
    isolation_window_target    DOUBLE,
    isolation_window_lower     DOUBLE,
    isolation_window_upper     DOUBLE,
    spectrum_ref               VARCHAR,

    base_peak_mz               DOUBLE,
    base_peak_intensity        DOUBLE,
    tic                        DOUBLE,
    lowest_mz                  DOUBLE,
    highest_mz                 DOUBLE,
    num_peaks                  BIGINT NOT NULL,

    polarity                   VARCHAR,
    centroided                 BOOLEAN,
    filter_string              VARCHAR,
    ion_injection_time         DOUBLE,
    ion_injection_time_unit    VARCHAR,
    monoisotopic_mz            DOUBLE,
    scan_window_lower          DOUBLE,
    scan_window_upper          DOUBLE
);
```

Column semantics:

| Column | Required | Semantics |
|---|---:|---|
| `scan_id` | yes | Contiguous mzDuck id assigned in import order, starting at 0. Used as the main join key. |
| `source_index` | yes | pyteomics mzML spectrum `index`, if available; otherwise the source iteration index. |
| `native_id` | yes | Original mzML spectrum `id`, for example `controllerType=0 controllerNumber=1 scan=37`. |
| `scan_number` | no | Integer parsed from `native_id` when a `scan=` token exists. |
| `ms_level` | yes | Must be 2 for v1 imported rows. |
| `rt` | yes | Retention time value as stored from the source unit. |
| `rt_unit` | yes | Normalized source unit string, `minute` or `second` when recognized. |
| `precursor_mz` | no | Preferred precursor m/z for search and MGF `PEPMASS`; normally selected ion m/z. |
| `precursor_charge` | no | Preferred precursor charge for search and MGF `CHARGE`; normally selected ion charge. |
| `precursor_intensity` | no | Preferred precursor intensity for MGF `PEPMASS`; normally selected ion peak intensity. |
| `selected_ion_mz` | no | mzML selected ion m/z. |
| `selected_ion_intensity` | no | mzML selected ion peak intensity. |
| `selected_ion_charge` | no | mzML selected ion charge state. |
| `collision_energy` | no | Collision energy numeric value when present. |
| `activation_type` | no | Short activation label such as `HCD`, `CID`, `ETD`. |
| `activation_cv` | no | JSON object containing the source activation CV names/accessions that were seen. |
| `isolation_window_target` | no | mzML isolation window target m/z. |
| `isolation_window_lower` | no | mzML lower offset, not absolute lower m/z. |
| `isolation_window_upper` | no | mzML upper offset, not absolute upper m/z. |
| `spectrum_ref` | no | mzML precursor `spectrumRef`, if present. |
| `base_peak_mz` | no | mzML base peak m/z. |
| `base_peak_intensity` | no | mzML base peak intensity. |
| `tic` | no | mzML total ion current. |
| `lowest_mz` | no | Lowest observed m/z reported by mzML, or computed from peak array if absent. |
| `highest_mz` | no | Highest observed m/z reported by mzML, or computed from peak array if absent. |
| `num_peaks` | yes | Number of rows in `peaks` for this `scan_id`. |
| `polarity` | no | `positive`, `negative`, or NULL. |
| `centroided` | no | TRUE for centroid spectra, FALSE for profile spectra, NULL if unknown. |
| `filter_string` | no | Vendor filter string, especially Thermo. |
| `ion_injection_time` | no | Injection time numeric value. |
| `ion_injection_time_unit` | no | Usually `millisecond` when known. |
| `monoisotopic_mz` | no | Thermo trailer monoisotopic m/z if present. |
| `scan_window_lower` | no | Lower scan window m/z. |
| `scan_window_upper` | no | Upper scan window m/z. |

Required v1 import rules:

- `scan_id` must be contiguous from 0 to `COUNT(*) - 1`.
- `source_index` and `native_id` must preserve the mzML values.
- `ms_level` must be 2 for every row written by the v1 converter.
- `rt` and `rt_unit` must be present.
- `num_peaks` must equal the number of matching rows in `peaks`.
- `precursor_mz`, `precursor_charge`, and `precursor_intensity` should mirror
  the selected ion fields when those are available.
- Missing optional mzML values must be stored as NULL, not placeholder numbers.

### 5.4 Table: `peaks`

One row per MS2 peak.

```sql
CREATE TABLE peaks (
    scan_id      BIGINT NOT NULL,
    peak_index   BIGINT NOT NULL,
    mz           DOUBLE NOT NULL,
    intensity    REAL NOT NULL,
    PRIMARY KEY (scan_id, peak_index)
);
```

Column semantics:

| Column | Required | Semantics |
|---|---:|---|
| `scan_id` | yes | Foreign key-like link to `spectra.scan_id`. |
| `peak_index` | yes | Zero-based position in the original mzML m/z/intensity arrays. |
| `mz` | yes | Peak m/z. Stored as `DOUBLE` even when the source is float32. |
| `intensity` | yes | Peak intensity. Stored as `REAL` for v1 unless source precision requires promotion. |

Rules:

- Preserve source peak order with `peak_index`.
- Do not assume the source peak array is sorted by m/z.
- Exporters may sort by m/z only when a target format or validation path expects
  sorted peaks. Otherwise they should use `ORDER BY peak_index`.
- `peak_index` must be contiguous from 0 to `num_peaks - 1` for each scan.
- Empty spectra are allowed only if the source mzML contains them. They must
  have `num_peaks = 0` and no `peaks` rows.
- m/z and intensity arrays must have equal length. Mismatch is a hard error.

### 5.5 Table: `run_metadata`

Key-value metadata for schema information, provenance, source metadata, and
round-trip support.

```sql
CREATE TABLE run_metadata (
    key      VARCHAR PRIMARY KEY,
    value    VARCHAR
);
```

Required keys:

| Key | Value |
|---|---|
| `schema_version` | `1` |
| `format_name` | `mzDuck` |
| `creator` | package and command name, for example `mzduck convert` |
| `mzduck_version` | package version string, or `0+unknown` during early development |
| `duckdb_version` | DuckDB runtime version |
| `python_version` | Python runtime version |
| `conversion_timestamp` | ISO-8601 UTC timestamp |
| `source_path` | Absolute path to source mzML when available |
| `source_filename` | Basename of source mzML |
| `source_size_bytes` | Source file size as decimal string |
| `source_sha256` | SHA-256 checksum, or empty/NULL if disabled for performance |
| `source_format` | `mzML` |
| `run_id` | mzML `<run id>` |
| `run_start_time` | mzML run `startTimeStamp`, if present |
| `instrument_config_ref` | mzML run default instrument configuration ref, if present |
| `rt_unit` | Dominant source RT unit, usually `minute` |
| `spectrum_count` | Number of imported spectra |
| `peak_count` | Number of imported peaks |
| `mz_array_dtype` | Source m/z dtype reported by numpy, for example `float32` |
| `intensity_array_dtype` | Source intensity dtype reported by numpy, for example `float32` |

Recommended XML snapshot keys:

| Key | Description |
|---|---|
| `cv_list_xml` | Raw `<cvList>` XML string. |
| `file_description_xml` | Raw `<fileDescription>` XML string. |
| `referenceable_param_groups_xml` | Raw `<referenceableParamGroupList>` XML string, if present. |
| `software_list_xml` | Raw `<softwareList>` XML string. |
| `instrument_configuration_xml` | Raw `<instrumentConfigurationList>` XML string. |
| `data_processing_xml` | Raw `<dataProcessingList>` XML string. |
| `source_file_list_xml` | Raw `<sourceFileList>` XML string, if extracted separately. |

The raw XML snapshots are for provenance and best-effort export guidance. V1
mzML export must be built through `psims`; it is not required to splice raw XML
verbatim into the output.

JSON metadata values must be valid JSON strings when the value represents a
structured object. Plain scalar values may be stored as plain strings.

### 5.6 Indexes

Create indexes after all rows are inserted:

```sql
CREATE INDEX idx_peaks_scan_id ON peaks(scan_id);
CREATE INDEX idx_peaks_mz ON peaks(mz);
CREATE INDEX idx_spectra_rt ON spectra(rt);
CREATE INDEX idx_spectra_precursor_mz ON spectra(precursor_mz);
CREATE INDEX idx_spectra_native_id ON spectra(native_id);
CREATE INDEX idx_spectra_scan_number ON spectra(scan_number);
```

Index purposes:

| Index | Query pattern |
|---|---|
| `idx_peaks_scan_id` | Retrieve a full spectrum by `scan_id`. |
| `idx_peaks_mz` | Product-ion or fragment m/z range searches. |
| `idx_spectra_rt` | Retention-time windows. |
| `idx_spectra_precursor_mz` | Precursor m/z range searches. |
| `idx_spectra_native_id` | Map mzML ids back to mzDuck spectra. |
| `idx_spectra_scan_number` | Search by parsed scan number. |

Writers should create indexes only after import because maintaining indexes
during bulk insert is slower.

### 5.7 Views

V1 may create convenience views, but no user-facing behavior may depend on them.
If added, recommended views are:

```sql
CREATE VIEW spectrum_peaks AS
SELECT
    s.scan_id,
    s.native_id,
    s.rt,
    s.precursor_mz,
    s.precursor_charge,
    p.peak_index,
    p.mz,
    p.intensity
FROM spectra s
JOIN peaks p ON p.scan_id = s.scan_id;
```

Views are not part of the required v1 compatibility contract unless this
document is updated.

## 6. mzML field extraction contract

The importer uses `pyteomics.mzml.MzML` for spectrum iteration and `lxml` for
header extraction.

### 6.1 Spectrum dictionary fields

Expected pyteomics keys for the canonical file:

```python
[
    'MSn spectrum',
    'base peak intensity',
    'base peak m/z',
    'centroid spectrum',
    'count',
    'defaultArrayLength',
    'highest observed m/z',
    'id',
    'index',
    'intensity array',
    'lowest observed m/z',
    'm/z array',
    'ms level',
    'positive scan',
    'precursorList',
    'scanList',
    'total ion current',
]
```

The implementation must not assume all files expose exactly these keys. It must
use safe accessors and apply the required/nullability rules from this spec.

### 6.2 Scan fields

Common scan extraction paths:

```python
scan = spectrum['scanList']['scan'][0]
scan['scan start time']
scan['filter string']
scan['ion injection time']
scan['[Thermo Trailer Extra]Monoisotopic M/Z:']
scan['scanWindowList']['scanWindow'][0]['scan window lower limit']
scan['scanWindowList']['scanWindow'][0]['scan window upper limit']
```

Rules:

- `scan start time` is required for v1.
- If the returned RT value is a pyteomics `unitfloat`, record both the numeric
  value and normalized unit.
- Recognized RT units are `minute`, `second`, `min`, `s`, `UO:0000031`, and
  `UO:0000010`. Normalize to `minute` or `second` where possible.
- If an RT unit is unknown but a value is present, store the unit string as
  observed and fail export to MGF unless a conversion rule exists.
- `filter string`, injection time, monoisotopic m/z, and scan window fields are
  optional.

### 6.3 Precursor fields

Common precursor extraction paths:

```python
precursor = spectrum['precursorList']['precursor'][0]
precursor['spectrumRef']
precursor['isolationWindow']['isolation window target m/z']
precursor['isolationWindow']['isolation window lower offset']
precursor['isolationWindow']['isolation window upper offset']
precursor['activation']['collision energy']
precursor['selectedIonList']['selectedIon'][0]['selected ion m/z']
precursor['selectedIonList']['selectedIon'][0]['charge state']
precursor['selectedIonList']['selectedIon'][0]['peak intensity']
```

Rules:

- V1 expects one precursor entry per MS2 spectrum. If multiple precursors are
  present, store the first in the core columns and record a warning or JSON
  metadata for the full precursor list in a future extension. Do not silently
  merge multiple precursors.
- `precursor_mz` should use selected ion m/z when present. If selected ion m/z
  is absent, use isolation target m/z and record the missing selected ion field
  as NULL.
- `precursor_charge` should use selected ion charge when present.
- `precursor_intensity` should use selected ion peak intensity when present.
- Isolation lower/upper columns store offsets, not absolute boundaries.
- `spectrum_ref` stores the raw mzML precursor `spectrumRef`.

### 6.4 Activation mapping

`activation_type` uses short labels for common activation terms:

| pyteomics key or CV name | mzDuck `activation_type` |
|---|---|
| `beam-type collision-induced dissociation` | `HCD` |
| `higher-energy collision-induced dissociation` | `HCD` |
| `collision-induced dissociation` | `CID` |
| `electron transfer dissociation` | `ETD` |
| `electron capture dissociation` | `ECD` |
| `infrared multiphoton dissociation` | `IRMPD` |
| `supplemental collision-induced dissociation` | `SID` |
| unknown term | original term string |

`activation_cv` should store a JSON object containing the activation keys,
accessions, and values observed in the source precursor activation dictionary.
This preserves information for improved reverse mapping later.

### 6.5 Native id and scan number

`native_id` is always the source mzML `spectrum['id']`.

`scan_number` is parsed only when a token matching `scan=<integer>` is present.
If parsing fails, store NULL. Do not use `scan_number` as the primary key because
not all mzML native ids use Thermo scan numbers and scan numbers may not be
contiguous.

`source_index` is the mzML spectrum index reported by pyteomics. `scan_id` is the
mzDuck contiguous id assigned to imported spectra.

## 7. Conversion algorithms

### 7.1 mzML to mzDuck

Public entry point:

```python
MzDuckFile.from_mzml(
    mzml_path,
    output_path,
    *,
    overwrite=False,
    batch_size=5000,
)
```

Algorithm:

1. Validate inputs.
   - `mzml_path` must exist and be readable.
   - `output_path` must not exist unless `overwrite=True`.
   - Parent directory of `output_path` must exist.
   - Required Python packages must import.
2. Open a new DuckDB database at `output_path`.
3. Create the v1 tables without secondary indexes.
4. Extract run metadata from the XML header with `lxml`.
   - Store raw XML snapshots for the recommended metadata keys.
   - Store provenance keys, Python/DuckDB/package versions, source path, source
     filename, source size, optional SHA-256, and conversion timestamp.
5. Iterate spectra with `pyteomics.mzml.MzML(mzml_path)`.
6. For each spectrum:
   - Verify `ms level == 2`; otherwise fail unless a future explicit filtering
     option is enabled.
   - Extract scan metadata.
   - Extract precursor metadata.
   - Extract `m/z array` and `intensity array`.
   - Validate equal array lengths.
   - Assign the next contiguous `scan_id`.
   - Append one `spectra` row to the current metadata batch.
   - Append one `peaks` row per point with source-order `peak_index`.
7. Flush batches every `batch_size` spectra.
   - Use DuckDB `executemany`, a registered Arrow table, or a registered pandas
     DataFrame for the current batch only.
   - Do not accumulate all spectra or all peaks for the full run in memory.
8. After all spectra are inserted:
   - update `spectrum_count`, `peak_count`, source dtype metadata, and any
     warning metadata;
   - create secondary indexes;
   - run validation queries;
   - close cleanly.

Batching guidance:

- `batch_size=5000` spectra is the default starting point.
- Peak batches should be built from per-spectrum numpy arrays and released after
  insertion.
- If memory use becomes high, reduce `batch_size` before changing the schema.

Required post-import validation queries:

```sql
SELECT COUNT(*) FROM spectra;
SELECT COUNT(*) FROM peaks;

SELECT COUNT(*)
FROM spectra s
LEFT JOIN (
    SELECT scan_id, COUNT(*) AS n
    FROM peaks
    GROUP BY scan_id
) p ON p.scan_id = s.scan_id
WHERE s.num_peaks != COALESCE(p.n, 0);

SELECT MIN(scan_id), MAX(scan_id), COUNT(*)
FROM spectra;

SELECT scan_id, COUNT(*) AS n, MAX(peak_index) AS max_peak_index
FROM peaks
GROUP BY scan_id
HAVING n != max_peak_index + 1;
```

The converter must fail if validation finds mismatches.

### 7.2 mzDuck to MGF

Public entry point:

```python
MzDuckFile.open(path).to_mgf(output_path)
```

Algorithm:

1. Open `.mzduck` read-only.
2. Validate schema version.
3. Query spectra ordered by `scan_id`.
4. For each spectrum, query peaks ordered by `peak_index` unless a sorted output
   option is added later.
5. Write an MGF block:

```text
BEGIN IONS
TITLE=<native_id>
PEPMASS=<precursor_mz> <precursor_intensity>
CHARGE=<precursor_charge>+
RTINSECONDS=<rt converted to seconds>
<mz1> <intensity1>
<mz2> <intensity2>
END IONS
```

MGF rules:

- `TITLE` must use `native_id`.
- `PEPMASS` must use `precursor_mz`; append precursor intensity only when it is
  not NULL.
- `CHARGE` must be omitted when charge is NULL. Positive charges use `n+`.
- `RTINSECONDS` must be written when RT is available and convertible.
- If `rt_unit = minute`, multiply by 60.
- If `rt_unit = second`, write the value directly.
- If RT unit is unrecognized, fail with a clear error unless the user supplies a
  conversion override in a future API.
- Peak lines should use enough significant digits to preserve stored values.

Reference validation command:

```bash
/data/p/bin/msconvert 1556259.mzML --mgf -o /data2/pub/proteome/PRIDE/protinsight/2019/07/PXD010154/temp/temp/
```

### 7.3 mzDuck to mzML

Public entry point:

```python
MzDuckFile.open(path).to_mzml(output_path)
```

V1 must use:

```python
from psims.mzml import MzMLWriter
```

Algorithm:

1. Open `.mzduck` read-only.
2. Validate schema version and required tables.
3. Load `run_metadata`.
4. Initialize `psims.mzml.MzMLWriter`.
5. Write controlled vocabularies, file description, software list, instrument
   configuration, and data processing as closely as psims allows.
6. Start one run using `run_id`, `run_start_time`, and instrument config
   metadata when available.
7. For each spectrum ordered by `scan_id`:
   - read spectrum metadata;
   - read peaks ordered by `peak_index`;
   - write an MS2 centroid spectrum;
   - include retention time and unit;
   - include polarity when known;
   - include base peak, TIC, and observed m/z bounds when available;
   - include scan window when available;
   - include precursor selected ion, charge, intensity, isolation window,
     activation type, collision energy, and `spectrum_ref` when available;
   - include filter string, ion injection time, and monoisotopic m/z when psims
     supports the relevant CV params.
8. Close the writer cleanly so mzML indexes and closing tags are valid.

Round-trip expectation:

- The output mzML must be readable by pyteomics.
- The output mzML should be readable by ProteoWizard where possible.
- XML bytes, attribute ordering, base64 byte layout, and exact header formatting
  are not expected to match the source.
- Supported semantic fields listed in the validation section must match within
  numeric tolerance.

### 8.1 Public class

```python
class MzDuckFile:

        *,
        overwrite=False,
        batch_size=5000,

        """Convert one mzML file into one mzDuck file and return an open handle."""
```

## 9. SQL query examples

Get one spectrum:

```sql
SELECT p.mz, p.intensity
FROM peaks p
WHERE p.scan_id = 42
ORDER BY p.peak_index;
```

Find all MS2 spectra for a precursor mass within 10 ppm:

```sql
SELECT *
FROM spectra
WHERE ABS(precursor_mz - 500.25) / 500.25 * 1e6 < 10
ORDER BY rt;
```

Extract a product-ion chromatogram:

```sql
SELECT s.rt, SUM(p.intensity) AS xic
FROM spectra s
JOIN peaks p ON p.scan_id = s.scan_id
WHERE p.mz BETWEEN 175.118 AND 175.120
GROUP BY s.rt
ORDER BY s.rt;
```

Find precursors containing a fragment:

```sql
SELECT DISTINCT s.precursor_mz, s.precursor_charge, s.rt, s.native_id
FROM spectra s
JOIN peaks p ON p.scan_id = s.scan_id
WHERE p.mz BETWEEN 175.118 AND 175.120
ORDER BY s.rt;
```

Charge distribution:

```sql
SELECT precursor_charge, COUNT(*) AS spectra
FROM spectra
GROUP BY precursor_charge
ORDER BY precursor_charge;
```

Activation distribution:

```sql
SELECT activation_type, COUNT(*) AS spectra
FROM spectra
GROUP BY activation_type
ORDER BY spectra DESC;
```

Retention-time window:

```sql
SELECT *
FROM spectra
WHERE rt BETWEEN 10.0 AND 20.0
ORDER BY rt;
```

## 10. Failure behavior

### 10.1 Hard errors

The converter must fail with a clear exception or CLI error for:

- missing input file;
- unreadable input file;
- existing output path without `overwrite=True` or `--overwrite`;
- unsupported file extension when the user requests mzML import;
- missing required `scan start time`;
- missing `m/z array` or `intensity array`;
- mismatched m/z and intensity array lengths;
- unsupported MS level in v1 when no explicit filtering option is enabled;
- unrecognized schema version when reading;
- failed post-import validation;
- output mzML requested when `psims` is not installed.

### 10.2 Nullable values

The importer must store NULL for absent optional values:

- precursor charge;
- precursor intensity;
- collision energy;
- activation type;
- isolation window values;
- base peak values;
- TIC;
- observed m/z bounds when they cannot be computed;
- filter string;
- injection time;
- monoisotopic m/z;
- scan windows;
- polarity;
- centroid/profile flag.

### 10.3 Warnings

The importer should record warnings in `run_metadata` as JSON under
`conversion_warnings` for:

- multiple precursor entries where only the first was stored in core columns;
- unknown activation terms;
- unknown RT unit that was stored but cannot be exported to MGF;
- optional header XML sections that could not be extracted;
- optional metadata fields missing from many spectra.

Warnings must not hide hard data-integrity errors.

## 14. Design defaults and assumptions

- This document is the source of truth for v1 until replaced by a newer design.
- The documentation update is the only change in the current planning step; it
  does not create package code or tests.
- `mzDuck/` is its own Git repository. Run Git status and diffs from
  `/data/p/xiaolong/mzPeak/mzDuck`.
- V1 is Python-first.
- V1 requires `psims` for mzML export.
- V1 stores one mzML run per `.mzduck`.
- V1 imports centroid MS2 spectra only.
- mzML round-trip means semantic equivalence for supported fields, not
  byte-identical XML.
- Retention time is stored in the source unit and converted only when a target
  format requires a specific unit.
- The canonical test file remains the primary integration fixture.
