# mzDuck V2 Compact-and-Lossless Redesign

Original date: 2026-04-16  
Updated for implementation: 2026-04-17

## Implemented direction

The implemented v2 architecture now uses a physical `mgf` MS2 payload table plus an optional `ms2_spectra` detail table.

This is the important change from the earlier draft:

- `mgf` is physical again
- `ms2_spectra` is detail-only
- `--ms2-mgf-only` and full/default outputs are structurally different again
- empty optional tables are omitted from final output
- `.mzML.gz` is supported
- parquet and parquet-zip output modes exist for CLI conversion

## Storage split

### `mgf`

Physical MS2 payload columns:

- `scan_number`
- `source_index`
- `rt`
- `precursor_mz`
- `precursor_intensity`
- `precursor_charge`
- `mz_array`
- `intensity_array`

### `ms2_spectra`

Physical MS2 detail columns:

- `scan_number`
- `instrument_configuration_ref`
- `collision_energy`
- `activation_type`
- `isolation_window_target`
- `isolation_window_lower`
- `isolation_window_upper`
- `precursor_scan_number`
- `base_peak_mz`
- `base_peak_intensity`
- `tic`
- `lowest_mz`
- `highest_mz`
- `ion_injection_time`
- `monoisotopic_mz`
- `scan_window_lower`
- `scan_window_upper`

Removed from stored MS2 detail:

- RT
- precursor payload
- arrays
- redundant `ms_level`
- stored `title`

## Text and extra-parameter policy

- `TITLE` is always derived
- `native_id` uses a run-level template when exact
- `spectrum_ref` uses `precursor_scan_number` plus a run-level template when exact
- `filter_string` uses per-run exact encoder detection
- if no exact filter-string rule exists, original strings are stored
- unmapped mzML params go into `spectrum_extra_params`

## Output modes

### Default

- `run_metadata`
- `mgf`
- `ms2_spectra`
- optional MS1 and higher MSn tables
- non-empty fallback tables

### `--ms2-only`

- `run_metadata`
- `mgf`
- `ms2_spectra`
- non-empty fallback tables

### `--ms2-mgf-only`

- `run_metadata`
- `mgf`

## Container outputs

### DuckDB `.mzduck`

- compact-copy finalization with `COPY FROM DATABASE`
- validated before rename

### `--parquet`

- folder output
- one parquet file per physical relation
- empty optional relations omitted

### `--parquet-zip`

- same parquet members inside one uncompressed zip archive

## Gzip support

Supported inputs:

- `.mzML`
- `.mzML.gz`

The implementation uses gzip-aware helpers for:

- header parsing
- provenance/header extraction
- mzML spectrum iteration

## Reader/export consequences

- `MzDuckFile.get_spectrum()` joins `mgf` with `ms2_spectra` when detail exists
- `export_mgf()` reads physical `mgf` and derives `TITLE`
- `export_mzml()` uses `mgf` for payload and source order, then overlays detail from `ms2_spectra` when present

## Compatibility

- new writes default to `schema_version = 2`
- readers still support v1 and v2
- top-level imports remain stable:
  `MzDuckFile`, `from_mzml`, `open`, `to_mgf`, `to_mzml`
