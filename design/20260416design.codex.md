# mzDuck V1 format and package specification

Date: 2026-04-16

Status: V1 implementation specification

This document is the development guide for the first mzDuck implementation. It
defines the `.mzduck` file contract, the Python package API, the command-line
interface, conversion algorithms, failure behavior, and validation plan. The
intent is that coding can start from this document without re-deciding the
architecture.

## 2026-04-17 implementation update

The implemented schema now supersedes the earlier single-`spectra` table draft
that appears in older sections below. The current v1 implementation must be
read as follows:

- mzDuck supports centroid mzML input with MS1 and MS2 spectra. MS3 and higher
  levels are reserved as separate `msN_spectra` tables and are not stored in
  the MGF contract.
- The `mgf` table is the MS2 MGF contract. It stores only fields used by MGF
  export: `scan_number`, `title`, `rt`, precursor m/z/intensity/charge, and
  `mz_array` / `intensity_array`.
- MS2 metadata that is not written directly to MGF must be stored in
  `ms2_spectra`, keyed by `scan_number`. This includes native id, source index,
  activation type, collision energy, isolation window, precursor spectrum
  reference, base peak, TIC, observed m/z bounds, filter string, ion injection
  time, monoisotopic m/z, and scan window bounds.
- MS1 spectra must be stored in `ms1_spectra`, with separate MS1 peak arrays.
  MS1 rows must not be mixed into `mgf`.
- Higher MSn levels must use separate tables named `ms3_spectra`,
  `ms4_spectra`, and so on when supported by default-mode import.
- `run_metadata` must contain mzML header fragments such as `fileDescription`,
  `referenceableParamGroupList`, `softwareList`,
  `instrumentConfigurationList`, `dataProcessingList`, and `sourceFileList`.
  It must also contain source counts, included counts, dtype/compression
  choices, total MS1/MS2/MSn summaries, and a JSON `table_registry` that
  describes which DuckDB tables are present and how they should be used.
- `spectrum_summary` is present in all modes except `--ms2-mgf-only`; it stores
  one row per included spectrum and helps quick inspection across MS levels.
- The default conversion mode creates `run_metadata`, `mgf`, `ms2_spectra`,
  `spectrum_summary`, and MS1/MSn tables when those levels are present.
- `--ms2-mgf-only` must create only `run_metadata` and `mgf`.
- `--ms2-only` must create `run_metadata`, `mgf`, `ms2_spectra`, and
  `spectrum_summary`.
- `--ms1-only` must create `run_metadata`, `ms1_spectra`, and
  `spectrum_summary`.
- `--no-ms1` skips MS1 in default mode.
- `--start-scan` and `--end-scan` filter selected spectra by inclusive
  `scan_number`.
- `--index-scan` creates exactly one optional index:
  `idx_mgf_scan_number ON mgf(scan_number)`. The older
  `--index-scan-number` spelling is retained only as a compatibility alias.
- MGF `TITLE` must be stored in `mgf.title` and exported verbatim. The default
  title format is `{mzduck_output_stem}.{scan_number}.{scan_number}.{charge}`;
  for example, `1556259.39.39.2`.
- mzML export must work from a full mzDuck file and from an `--ms2-mgf-only`
  file. When only `mgf` is available, mzML export writes minimal valid MS2
  spectra from the MGF contract; when detail tables are available, it uses them
  for richer semantic mzML reconstruction.

## 1. Summary

mzDuck stores centroid MS2 mzML data in a single DuckDB database file. The v1
format is intentionally relational, but peaks live as per-spectrum list
columns on the `spectra` row (not one row per peak) so the `.mzduck` file
size stays close to mzMLb:

- one wide `spectra` table holds per-scan metadata plus the peak arrays
  (`mz_array`, `intensity_array`) for that scan;
- row identity is the mzML-native `scan_number`; no mzDuck surrogate keys
  are invented;
- run-constant fields (`rt_unit`, `polarity`, `centroided`,
  `ion_injection_time_unit`) and a `native_id_template` are hoisted into
  `run_metadata`, so they are stored once per run, not once per row;
- a `peaks` view exposes the flat `(scan_number, peak_index, mz,
  intensity)` shape on top of `spectra.mz_array` / `intensity_array` for
  familiar SQL;
- run-level and provenance metadata is stored in `run_metadata`;
- v1 creates **zero** secondary indexes by default, matching mzMLb's
  indexless HDF5 layout. An opt-in `--index-scan-number` flag creates a
  single ART index for workloads that need it.

File size target: `.mzduck` must be close to the matching mzMLb for the same
run. The reference fixture `1556259.mzML` is 428 MB and `1556259.mzMLb` is
83 MB; the v1 `.mzduck` is expected in the 85-115 MB range. An early draft
of v1 stored one row per peak plus an m/z B-tree and produced a 1.2 GB
`.mzduck`; that design is explicitly rejected here.

The v1 Python package is named `mzduck`. It must support:

- mzML -> mzDuck;
- mzDuck -> MGF;
- mzDuck -> mzML;
- direct SQL queries against the `.mzduck` file;
- programmatic spectrum retrieval by `scan_number`.

The v1 mzML writer must use `psims.mzml.MzMLWriter`. Exported mzML files must be
semantically equivalent to the source for supported MS2 spectra, but they are
not expected to be byte-identical to the input XML.

## 2. Goals, non-goals, and scope

### 2.1 Goals

V1 must provide a practical development-quality format and package for MS2
proteomics data:

1. Use one portable `.mzduck` DuckDB file per mzML run.
2. Store enough MS2 spectrum metadata to reconstruct useful mzML and MGF output.
3. Make common MS2 queries simple SQL, whether over `spectra` metadata or over
   the `peaks` view that unnests the per-row peak arrays.
4. Avoid the full-run in-memory DataFrame pattern used by early mzsql scripts.
5. Preserve source ordering, source identifiers, and source dtype so exported
   files remain traceable to the original mzML and byte-equivalent for the
   peak arrays.
6. Keep the `.mzduck` file size close to the matching mzMLb for the same run.
7. Make the schema explicit enough for future readers in Python, R, or Rust.

### 2.2 V1 supported input

V1 supports mzML files with centroid MS2 spectra. The canonical test input is:

```text

```

| MS2 spectra | 55,814 |

| Average peaks per spectrum | 247.6 |

| intensity array dtype | float32 |

| Polarity | positive |

The same directory also contains `1556259.mzMLb` (84 MB). mzMLb is useful for

### 2.3 V1 outputs

V1 must write:

- `.mzduck` DuckDB database files;
- `.mgf` files;
- `.mzML` files written through `psims`.

### 2.4 V1 non-goals

V1 does not support:

- MS1 import or export as a required feature;
- chromatograms;
- wavelength spectra;
- profile-mode arrays;
- mzMLb input;
- vendor RAW input;
- multiple mzML files in one `.mzduck`;
- multi-run mzML files;
- byte-identical mzML reconstruction;
- lossless preservation of every arbitrary vendor XML extension.

Unsupported content must not be silently misrepresented. The converter must
either reject unsupported content with a clear error, or skip it only when an
explicit user option requests filtering.

### 2.5 Future extension points

The schema is allowed to evolve to support MS1 later by adding rows to the
same `spectra` table with `ms_level = 1` and nullable precursor fields. V1
code must not rely on every future spectrum having precursor data. However,
the v1 converter should fail on unexpected MS1 content unless the user
explicitly chooses an MS2-only import mode.

Chromatograms and wavelength spectra should be added later as separate
tables, not overloaded into the `spectra` peak-array columns.

## 3. Background and prior art

### 3.1 mzPeak limitations motivating mzDuck

The sibling mzPeak project stores data as an uncompressed ZIP archive containing
multiple Parquet files plus a JSON index. This design is rich and round-trips
well through the Rust implementation, but it is not ideal for simple SQL use:

- data appears as nested struct/list columns in DuckDB and pandas;
- random access depends on Parquet page statistics and external metadata;
- metadata tables are complex because they model broad mzML/mzdata concepts;
- users need custom readers instead of opening one relational database file.

mzDuck chooses a narrower scope and a flatter relational model.

### 3.2 mzsql prior art

The sibling mzsql project already converts mzML files into DuckDB/SQLite tables:

- `MS1(filename, id, mz, int, rt)`;
- `MS2(filename, id, premz, fragmz, int, rt)`;
- `file_info(filename, n_scans, rt_start, rt_end)`;
- `scan_info(filename, rt, scan_num, min_mz, max_mz, ms_level, centroided, polarity)`.

That model is excellent for quick range queries, but it does not store enough
metadata for meaningful mzML round-trip. Missing fields include charge state,
precursor intensity, activation type, collision energy, isolation window,
base peak, TIC, filter string, injection time, scan windows, native spectrum id,
precursor spectrum reference, and run-level metadata.

mzDuck keeps the queryability of mzsql while storing the MS2 metadata needed for
MGF and mzML export.

## 5. mzDuck file contract

A `.mzduck` file is a DuckDB database file. It must be valid DuckDB and must
contain the tables and indexes defined below.

V1 supports exactly one mzML run per `.mzduck` file. The source mzML basename,
absolute source path when available, and source run id are stored in
`run_metadata`.

### 5.1 Naming and compatibility

- File extension: `.mzduck`.
- Schema version key: `run_metadata['schema_version'] = '1'`.
- Package code must reject newer major schema versions unless it explicitly
  knows how to read them.
- Writers must create all v1 tables, even if optional columns contain only NULL.
- Readers must tolerate additional future columns but must require the v1
  columns listed here.

### 5.2 Type conventions

Use DuckDB SQL types:

| Logical value | DuckDB type | Notes |
|---|---|---|
| IDs and counts | `BIGINT` | Use `BIGINT` for future large files. |
| MS level and charge | `INTEGER` | Charge may be NULL. |
| m/z | `DOUBLE` | Preserve precision for calculations. |
| intensity | `REAL` by default | Promote only if source requires it. |
| retention time | `DOUBLE` | Stored in source unit. |
| text | `VARCHAR` | Use NULL for absent values. |
| booleans | `BOOLEAN` | Use NULL when unknown. |
| timestamps | `VARCHAR` | ISO-8601 string in metadata. |
| structured metadata | `VARCHAR` | JSON string. |

DuckDB does not enforce all foreign keys efficiently for this use case. The
writer must enforce invariants in code and validate them after import.

### 5.3 Table: `spectra`

One row per imported MS2 spectrum. Every column is chosen to live in the
mzML source itself (no mzDuck-invented surrogate keys) and to use the
smallest DuckDB type that still round-trips losslessly. Peak arrays are on
the same row; there is no separate `spectrum_arrays` table.

```sql
CREATE TABLE spectra (
    -- identity
    scan_number                INTEGER   NOT NULL,
    native_id                  VARCHAR,                          -- NULL when native_id_template covers this row

    -- core scan
    ms_level                   UTINYINT  NOT NULL,
    rt                         FLOAT     NOT NULL,

    -- precursor
    precursor_mz               DOUBLE,
    precursor_charge           TINYINT,
    precursor_intensity        FLOAT,
    collision_energy           FLOAT,
    activation_type            VARCHAR,
    isolation_window_target    DOUBLE,
    isolation_window_lower     FLOAT,
    isolation_window_upper     FLOAT,
    spectrum_ref               VARCHAR,

    -- spectrum summary
    base_peak_mz               FLOAT,
    base_peak_intensity        FLOAT,
    tic                        FLOAT,
    lowest_mz                  FLOAT,
    highest_mz                 FLOAT,

    -- per-scan extras
    filter_string              VARCHAR,
    ion_injection_time         FLOAT,
    monoisotopic_mz            DOUBLE,
    scan_window_lower          FLOAT,
    scan_window_upper          FLOAT,

    -- peak arrays (merged here; see 5.4 for dtype rules)
    mz_array                   FLOAT[]   NOT NULL,
    intensity_array            FLOAT[]   NOT NULL
);
```

#### 5.3.1 Dropped columns (and why)

| First-draft column | Disposition | Reason |
| --- | --- | --- |
| `scan_id` | removed | mzDuck-invented contiguous surrogate. Not present in mzML. `scan_number` serves as the identifier instead. |
| `source_index` | removed | Mirror of the mzML `<spectrum index>` attribute, which is positional. Reconstructable from output ordering. |
| `selected_ion_mz` / `selected_ion_intensity` / `selected_ion_charge` | removed | Redundant: `precursor_mz`, `precursor_intensity`, `precursor_charge` already carry these values for v1. |
| `activation_cv` | removed | `activation_type` + `collision_energy` are sufficient for v1 activation round-trip. |
| `num_peaks` | removed | Derivable in O(1) from `len(mz_array)` at query time. |
| `rt_unit` | moved to `run_metadata` | Uniform per run in every observed file. Mixed values rejected at import. |
| `polarity` | moved to `run_metadata` | Uniform per run. |
| `centroided` | moved to `run_metadata` | Uniform per run. |
| `ion_injection_time_unit` | moved to `run_metadata` | Uniform per run. |

#### 5.3.2 Column semantics

| Column | Required | Semantics |
| --- | ---: | --- |
| `scan_number` | yes | Integer parsed from the `scan=<N>` token in `native_id`. If no such token exists, falls back to the mzML `<spectrum index>` attribute. Used as the user-facing identifier. |
| `native_id` | no | Original mzML spectrum `id` string. NULL when `run_metadata.native_id_template` is set and formatting the template with this row's `scan_number` reproduces the original value. |
| `ms_level` | yes | Must be 2 for every v1 imported row. |
| `rt` | yes | Retention time in `run_metadata.rt_unit` (typically `minute`). |
| `precursor_mz` | no | Preferred precursor m/z for search and MGF `PEPMASS`; normally the selected ion m/z. |
| `precursor_charge` | no | Preferred precursor charge for search and MGF `CHARGE`; normally the selected ion charge. |
| `precursor_intensity` | no | Preferred precursor intensity for MGF `PEPMASS`. |
| `collision_energy` | no | Collision energy numeric value when present. |
| `activation_type` | no | Short activation label such as `HCD`, `CID`, `ETD`. |
| `isolation_window_target` | no | mzML isolation window target m/z. |
| `isolation_window_lower` | no | mzML lower offset, not absolute lower m/z. |
| `isolation_window_upper` | no | mzML upper offset, not absolute upper m/z. |
| `spectrum_ref` | no | mzML precursor `spectrumRef`, if present. |
| `base_peak_mz` | no | mzML base peak m/z. |
| `base_peak_intensity` | no | mzML base peak intensity. |
| `tic` | no | mzML total ion current. |
| `lowest_mz` | no | Lowest observed m/z, copied from mzML or computed from the peak array. Used for XIC pruning. |
| `highest_mz` | no | Highest observed m/z. Used for XIC pruning. |
| `filter_string` | no | Vendor filter string, especially Thermo. |
| `ion_injection_time` | no | Injection time value in `run_metadata.ion_injection_time_unit`. |
| `monoisotopic_mz` | no | Thermo trailer monoisotopic m/z if present. |
| `scan_window_lower` | no | Lower scan window m/z. |
| `scan_window_upper` | no | Upper scan window m/z. |
| `mz_array` | yes | Source-order m/z values. Element dtype per 5.4. |
| `intensity_array` | yes | Source-order intensity values. Element dtype per 5.4. |

#### 5.3.3 Type rationale

| Column(s) | DuckDB type | Why |
| --- | --- | --- |
| `scan_number` | `INTEGER` (INT32) | Signed 32-bit covers every real instrument. |
| `ms_level` | `UTINYINT` | 1 byte. Values in 1-10. |
| `rt` | `FLOAT` | 4 bytes. ~0.4 ms at 60 min, far below chromatographic scale. |
| `precursor_charge` | `TINYINT` | 1 byte. Signed so negative-mode charges remain honest. |
| `precursor_mz`, `isolation_window_target`, `monoisotopic_mz` | `DOUBLE` | Downstream identification tolerates float32 but m/z precision is the one field users care about most. |
| `precursor_intensity`, `collision_energy`, `base_peak_mz`, `base_peak_intensity`, `tic`, `lowest_mz`, `highest_mz`, `isolation_window_lower`, `isolation_window_upper`, `ion_injection_time`, `scan_window_lower`, `scan_window_upper` | `FLOAT` | The mzML source is float32 for these fields. |
| `activation_type`, `native_id`, `spectrum_ref`, `filter_string` | `VARCHAR` | DuckDB FSST + dictionary dedups shared prefixes automatically. |

#### 5.3.4 Required v1 import rules

- `scan_number` must be present for every row. Falls back to the mzML
  `<spectrum index>` attribute when no `scan=<N>` token is parseable.
- `ms_level` must be 2 for every row written by the v1 converter.
- `rt` must be present. If the source unit disagrees with
  `run_metadata.rt_unit`, importing fails.
- The importer must assert `run_metadata.polarity`, `centroided`, and
  `ion_injection_time_unit` are uniform across the run. Mixed values are a
  hard error.
- `precursor_mz`, `precursor_charge`, and `precursor_intensity` copy from
  the selected ion fields when those are available; they are NULL
  otherwise.
- Missing optional mzML values are stored as NULL, not placeholder
  numbers.
- `mz_array` and `intensity_array` must have equal length for every row.
- `native_id` is NULL iff `run_metadata.native_id_template` can reproduce
  the original id for this row's `scan_number`.

### 5.4 Peak arrays (embedded in `spectra`)

Peak arrays live on the `spectra` row as `mz_array` and `intensity_array`.
There is no separate `spectrum_arrays` table; the first-draft v1 had one
and it is deleted here because the relationship to `spectra` is strictly
1:1, a join adds nothing, and DuckDB's column-pruned scans make
metadata-only queries equally cheap either way.

#### 5.4.1 Array semantics

| Column | Required | Semantics |
| --- | ---: | --- |
| `mz_array` | yes | m/z values in source order. List element type mirrors the source numpy dtype. |
| `intensity_array` | yes | Intensity values in source order. List element type mirrors the source numpy dtype. |

Rules:

- The two arrays must have equal length. Mismatch is a hard error at
  import.
- Element order matches the mzML source; do not reorder by m/z at import.
  Exporters may reorder only when the target format demands it.
- Empty spectra are allowed if the source mzML contains them, represented
  as two zero-length lists.

#### 5.4.2 Storage dtype

The importer picks the list element type once per `.mzduck` file, for
each of m/z and intensity independently, based on the source numpy dtype
of every spectrum's arrays:

| Observed source dtype | Stored DuckDB type | Notes |
| --- | --- | --- |
| every spectrum `float32` | `FLOAT[]` | Lossless. DuckDB `FLOAT` is IEEE 754 binary32, same as mzML `MS:1000521`. |
| any spectrum `float64` | `DOUBLE[]` | Required to avoid truncation. |
| exotic ints or other | `DOUBLE[]` | Cast once at import; log a warning. |

The final per-array choice is recorded in `run_metadata` under
`mz_array_storage_dtype` and `intensity_array_storage_dtype` so the mzML
exporter emits the correct `MS:1000521` / `MS:1000523` CV terms and
reproduces the original binary array layout.

Do not store the source dtype in the table itself: DuckDB list columns
are uniform per column, and a mixed-dtype column would force the widest
type on every row.

#### 5.4.3 Why per-spectrum arrays and not one row per peak

Storing one row per `(scan_id, peak_index, mz, intensity)` is the
obvious relational model, but it wastes enormous space for MS data.
Measured on the canonical fixture (55,814 spectra, 13,819,715 peaks):

| Schema | `.mzduck` size |
| --- | ---: |
| `peaks` row-per-peak with `(BIGINT, BIGINT, DOUBLE, REAL)` and full indexes | ~1,230 MB |
| `spectra` with embedded `FLOAT[]` arrays, no secondary indexes | ~91 MB |
| Reference `1556259.mzMLb` (HDF5 + zlib) | 83 MB |
| Reference `1556259.mzML` (indexed mzML with zlib binary arrays) | 408 MB |

The row-per-peak format pays three costs that the array format avoids:

1. `scan_id` is written 13.8 million times rather than 55,814 times.
2. `peak_index` is written 13.8 million times even though it is
   positional.
3. A full-peak-universe m/z B-tree index (`idx_peaks_mz`) is needed to
   make product-ion range queries tolerable; on 13.8M effectively-random
   doubles this index alone is hundreds of megabytes.

Storing peaks as list columns also aligns naturally with how the source
mzML stores them (two base64 float arrays per spectrum) and with how
pyteomics returns them (two numpy arrays per spectrum), so import and
export avoid an extra flatten / regroup step.

#### 5.4.4 `peaks` compatibility view

The v1 writer creates a view so users can still write familiar
peak-level SQL:

```sql
CREATE VIEW peaks AS
SELECT
    scan_number,
    generate_subscripts(mz_array, 1) - 1 AS peak_index,
    UNNEST(mz_array)        AS mz,
    UNNEST(intensity_array) AS intensity
FROM spectra;
```

Rules:

- The view is part of the v1 compatibility contract.
- Users may rely on columns `scan_number`, `peak_index`, `mz`,
  `intensity`.
- The view is unordered; queries that need ordered output must use
  `ORDER BY scan_number, peak_index`.
- Single-spectrum reads should query `spectra` directly (`SELECT
  mz_array, intensity_array FROM spectra WHERE scan_number = 42`) to
  avoid the UNNEST round-trip.

### 5.5 Table: `run_metadata`

Key-value metadata for schema information, provenance, source metadata, and
round-trip support.

```sql
CREATE TABLE run_metadata (
    key      VARCHAR PRIMARY KEY,
    value    VARCHAR
);
```

Required keys:

| Key | Value |
| --- | --- |
| `schema_version` | `1` |
| `format_name` | `mzDuck` |
| `creator` | package and command name, for example `mzduck convert` |
| `mzduck_version` | package version string, or `0+unknown` during early development |
| `duckdb_version` | DuckDB runtime version |
| `python_version` | Python runtime version |
| `conversion_timestamp` | ISO-8601 UTC timestamp |
| `source_path` | Absolute path to source mzML when available |
| `source_filename` | Basename of source mzML |
| `source_size_bytes` | Source file size as decimal string |
| `source_sha256` | SHA-256 checksum, or empty/NULL if disabled for performance |
| `source_format` | `mzML` |
| `run_id` | mzML `<run id>` |
| `run_start_time` | mzML run `startTimeStamp`, if present |
| `instrument_config_ref` | mzML run default instrument configuration ref, if present |
| `rt_unit` | Run-constant source RT unit. Must be uniform across the run. Typical value `minute`. |
| `polarity` | Run-constant polarity. One of `positive` or `negative`. |
| `centroided` | Run-constant string `true` or `false`. v1 accepts only `true`. |
| `ion_injection_time_unit` | Run-constant injection-time unit. Typical value `millisecond`. |
| `native_id_template` | Python-style format string that reconstructs the full mzML `spectrum id` from `scan_number`, for example `controllerType=0 controllerNumber=1 scan={scan_number}`. Set only when every source id matches the same template. |
| `spectrum_count` | Number of imported spectra |
| `peak_count` | Total peaks (sum of `len(mz_array)`) |
| `mz_array_dtype` | Source m/z dtype reported by numpy, for example `float32` |
| `intensity_array_dtype` | Source intensity dtype reported by numpy, for example `float32` |
| `mz_array_storage_dtype` | Stored DuckDB list element type for `spectra.mz_array`, one of `FLOAT` or `DOUBLE` |
| `intensity_array_storage_dtype` | Stored DuckDB list element type for `spectra.intensity_array`, one of `FLOAT` or `DOUBLE` |
| `compression` | Forced DuckDB compression mode used at import, for example `zstd` |
| `compression_level` | Requested zstd compression level as a decimal string |
| `index_scan_number` | `true` if `idx_spectra_scan_number` was created, `false` otherwise. Default `false`. |

Recommended XML snapshot keys:

| Key | Description |
|---|---|
| `cv_list_xml` | Raw `<cvList>` XML string. |
| `file_description_xml` | Raw `<fileDescription>` XML string. |
| `referenceable_param_groups_xml` | Raw `<referenceableParamGroupList>` XML string, if present. |
| `software_list_xml` | Raw `<softwareList>` XML string. |
| `instrument_configuration_xml` | Raw `<instrumentConfigurationList>` XML string. |
| `data_processing_xml` | Raw `<dataProcessingList>` XML string. |
| `source_file_list_xml` | Raw `<sourceFileList>` XML string, if extracted separately. |

The raw XML snapshots are for provenance and best-effort export guidance. V1
mzML export must be built through `psims`; it is not required to splice raw XML
verbatim into the output.

JSON metadata values must be valid JSON strings when the value represents a
structured object. Plain scalar values may be stored as plain strings.

### 5.6 Indexes

**v1 creates zero secondary indexes by default.** This matches the indexless
HDF5 layout of mzMLb and keeps the `.mzduck` file size close to the 83 MB
mzMLb reference for the canonical fixture. `scan_number` is not declared as
`PRIMARY KEY`, because even a `PRIMARY KEY` constraint materializes an ART
index block that inflates the output file without helping v1 workloads at
55k-spectrum scale.

Optional opt-in index, created only when the user asks for it:

```sql
-- Created only when convert --index-scan-number is passed.
CREATE INDEX idx_spectra_scan_number ON spectra(scan_number);
```

Rationale:

- DuckDB 1.5 keeps per-column zonemaps (min/max) on every row group. For
  numeric columns like `rt`, `precursor_mz`, `lowest_mz`, `highest_mz`,
  range predicates already prune row groups without a B-tree.
- `native_id` is usually reconstructed from `run_metadata.native_id_template`
  plus `scan_number`, so explicit `native_id` lookups are rare.
- A search by exact `scan_number` on a 55k-row column is a ~1 MB table
  scan; the index only pays off for workloads that issue many such
  single-row lookups against very large files.

The first draft of this document also proposed `idx_spectra_rt`,
`idx_spectra_precursor_mz`, `idx_spectra_native_id`, `idx_peaks_scan_id`,
and `idx_peaks_mz`. All are dropped in v1:

- `idx_peaks_scan_id` / `idx_peaks_mz`: there is no peaks table; the
  per-peak m/z index on 13.8M random doubles was the single largest
  contributor to the ~1.2 GB first draft and product-ion range queries
  should prune on `spectra.lowest_mz` / `spectra.highest_mz` before any
  UNNEST.
- `idx_spectra_rt`, `idx_spectra_precursor_mz`: zonemaps already handle
  these range queries acceptably.
- `idx_spectra_native_id`: redundant with `native_id_template` +
  `scan_number`.

Index creation rules:

- If `--index-scan-number` was passed, create `idx_spectra_scan_number`
  **after** all rows are inserted, then set
  `run_metadata.index_scan_number = 'true'`.
- Run `CHECKPOINT` after any index creation so the forced compression
  (see section 5.8) is applied to the final blocks.
- Writers must not create any other secondary index in v1.

### 5.7 Views

V1 requires the `peaks` compatibility view (section 5.4.4). Additional
convenience views are optional.

Recommended optional view joining metadata and peaks for analytics:

```sql
CREATE VIEW spectrum_peaks AS
SELECT
    s.scan_number,
    s.rt,
    s.precursor_mz,
    s.precursor_charge,
    generate_subscripts(s.mz_array, 1) - 1 AS peak_index,
    UNNEST(s.mz_array)        AS mz,
    UNNEST(s.intensity_array) AS intensity
FROM spectra s;
```

Only `peaks` is part of the required v1 compatibility contract.
`spectrum_peaks` is optional.

### 5.8 Storage compression

mzDuck forces DuckDB's zstd column compressor by default. The CLI and Python
API both expose the knob so users can trade size for speed on the
write side:

- `--compression` / `compression`, one of `{zstd, auto, uncompressed}`,
  default `zstd`. Applied via `SET force_compression = '<value>'` on the
  output connection before any data is written.
- `--compression-level` / `compression_level`, integer, default `6`. Passed
  through to DuckDB if the running build accepts it; otherwise the
  converter records a single warning entry in
  `run_metadata.conversion_warnings` and proceeds with DuckDB's default
  zstd level. Kept as a stable CLI knob across DuckDB upgrades.

Rules:

- The importer must set the compression mode before any DDL or inserts
  targeting `spectra` or `run_metadata` run.
- The importer must execute `CHECKPOINT` after all inserts and any
  optional index creation so the forced compression is applied to the
  final blocks.
- The chosen compression mode and level must be recorded in `run_metadata`
  under `compression` and `compression_level` for provenance.
- On the canonical fixture, `zstd` and `auto` produce near-identical
  `.mzduck` sizes because the dominant column type is float and DuckDB's
  ALP / Patas / Chimp codecs already compete with zstd on floats. The knob
  is mostly valuable for the text metadata columns.

## 6. mzML field extraction contract

The importer uses `pyteomics.mzml.MzML` for spectrum iteration and `lxml` for
header extraction.

### 6.1 Spectrum dictionary fields

Expected pyteomics keys for the canonical file:

```python
[
    'MSn spectrum',
    'base peak intensity',
    'base peak m/z',
    'centroid spectrum',
    'count',
    'defaultArrayLength',
    'highest observed m/z',
    'id',
    'index',
    'intensity array',
    'lowest observed m/z',
    'm/z array',
    'ms level',
    'positive scan',
    'precursorList',
    'scanList',
    'total ion current',
]
```

The implementation must not assume all files expose exactly these keys. It must
use safe accessors and apply the required/nullability rules from this spec.

### 6.2 Scan fields

Common scan extraction paths:

```python
scan = spectrum['scanList']['scan'][0]
scan['scan start time']
scan['filter string']
scan['ion injection time']
scan['[Thermo Trailer Extra]Monoisotopic M/Z:']
scan['scanWindowList']['scanWindow'][0]['scan window lower limit']
scan['scanWindowList']['scanWindow'][0]['scan window upper limit']
```

Rules:

- `scan start time` is required for v1.
- If the returned RT value is a pyteomics `unitfloat`, record both the numeric
  value and normalized unit.
- Recognized RT units are `minute`, `second`, `min`, `s`, `UO:0000031`, and
  `UO:0000010`. Normalize to `minute` or `second` where possible.
- If an RT unit is unknown but a value is present, store the unit string as
  observed and fail export to MGF unless a conversion rule exists.
- `filter string`, injection time, monoisotopic m/z, and scan window fields are
  optional.

### 6.3 Precursor fields

Common precursor extraction paths:

```python
precursor = spectrum['precursorList']['precursor'][0]
precursor['spectrumRef']
precursor['isolationWindow']['isolation window target m/z']
precursor['isolationWindow']['isolation window lower offset']
precursor['isolationWindow']['isolation window upper offset']
precursor['activation']['collision energy']
precursor['selectedIonList']['selectedIon'][0]['selected ion m/z']
precursor['selectedIonList']['selectedIon'][0]['charge state']
precursor['selectedIonList']['selectedIon'][0]['peak intensity']
```

Rules:

- V1 expects one precursor entry per MS2 spectrum. If multiple precursors are
  present, store the first in the core columns and record a warning or JSON
  metadata for the full precursor list in a future extension. Do not silently
  merge multiple precursors.
- `precursor_mz` should use selected ion m/z when present. If selected ion m/z
  is absent, use isolation target m/z and record the missing selected ion field
  as NULL.
- `precursor_charge` should use selected ion charge when present.
- `precursor_intensity` should use selected ion peak intensity when present.
- Isolation lower/upper columns store offsets, not absolute boundaries.
- `spectrum_ref` stores the raw mzML precursor `spectrumRef`.

### 6.4 Activation mapping

`activation_type` uses short labels for common activation terms:

| pyteomics key or CV name | mzDuck `activation_type` |
|---|---|
| `beam-type collision-induced dissociation` | `HCD` |
| `higher-energy collision-induced dissociation` | `HCD` |
| `collision-induced dissociation` | `CID` |
| `electron transfer dissociation` | `ETD` |
| `electron capture dissociation` | `ECD` |
| `infrared multiphoton dissociation` | `IRMPD` |
| `supplemental collision-induced dissociation` | `SID` |
| unknown term | original term string |

`activation_cv` should store a JSON object containing the activation keys,
accessions, and values observed in the source precursor activation dictionary.
This preserves information for improved reverse mapping later.

### 6.5 Native id and scan number

`scan_number` is the user-facing identifier in v1. It is parsed from the
source mzML `spectrum['id']`:

- If a `scan=<integer>` token is present, use that integer.
- Otherwise, fall back to the mzML `<spectrum index>` attribute reported
  by pyteomics.
- `scan_number` must be populated for every row. It is NOT guaranteed to
  be contiguous; a Thermo run that skips filtered MS1 scans will have
  gaps.

`native_id_template` handling:

- Before inserting rows, the importer checks whether every source id
  matches a single Python-style template with only `{scan_number}`
  varying, for example `controllerType=0 controllerNumber=1
  scan={scan_number}`.
- If so, the template is stored in `run_metadata.native_id_template` and
  every row's `native_id` column is written as NULL. The full id is
  reconstructed on export by formatting the template with `scan_number`.
- If the ids do not fit a single template, `run_metadata.native_id_template`
  is left NULL and every row stores its full `native_id` string verbatim.

`scan_number` is not declared as `PRIMARY KEY` in v1 (see 5.6) to avoid
the DuckDB ART block that a primary-key constraint materializes.

## 7. Conversion algorithms

### 7.1 mzML to mzDuck

Public entry point:

```python
MzDuckFile.from_mzml(
    mzml_path,
    output_path,
    *,
    overwrite=False,
    batch_size=5000,
    compression="zstd",          # {"zstd", "auto", "uncompressed"}
    compression_level=6,
    index_scan_number=False,
    compute_sha256=True,
)
```

Algorithm:

1. Validate inputs.
   - `mzml_path` must exist and be readable.
   - `output_path` must not exist unless `overwrite=True`.
   - Parent directory of `output_path` must exist.
   - Required Python packages must import.
   - `compression` must be one of the supported values.
   - `compression_level` must be a non-negative integer.
2. Open a new DuckDB database at `output_path`.
3. Apply compression settings *before* any DDL or inserts:
   - `SET force_compression = '<compression>'`.
   - Pass `compression_level` through to DuckDB where supported; record a
     `conversion_warnings` entry when the running DuckDB rejects it.
4. Single-pass pre-scan of the mzML header + lxml iterparse to resolve
   run-constant invariants before any DDL:
   - RT unit for every scan → single `rt_unit` value or abort with a
     mixed-unit error;
   - polarity for every scan → single `polarity` value or abort;
   - centroid/profile flag for every scan → single `centroided` value (v1
     demands `true`);
   - ion-injection-time unit for every scan → single
     `ion_injection_time_unit` value or abort;
   - source dtype of every `m/z array` and `intensity array` → pick
     `FLOAT[]` when all float32, else `DOUBLE[]`;
   - whether every `spectrum id` matches one Python-style template with
     only `{scan_number}` varying → capture `native_id_template` or
     leave NULL.
5. Create the v1 tables (`spectra`, `run_metadata`) and the `peaks`
   view. Do not create any secondary index yet.
6. Extract run metadata from the XML header with `lxml` and populate
   `run_metadata`:
   - raw XML snapshots for the recommended keys;
   - provenance: Python/DuckDB/package versions, source path, source
     filename, source size, optional SHA-256, conversion timestamp;
   - run-constant invariants captured in step 4: `rt_unit`, `polarity`,
     `centroided`, `ion_injection_time_unit`, `native_id_template`;
   - `compression`, `compression_level`, `mz_array_storage_dtype`,
     `intensity_array_storage_dtype`;
   - `index_scan_number` set to `'true'` or `'false'` based on the CLI
     flag.
7. Iterate spectra with `pyteomics.mzml.MzML(mzml_path)`.
8. For each spectrum:
   - Verify `ms level == 2`; otherwise fail unless a future explicit
     filtering option is enabled.
   - Parse `scan_number` from the `scan=<N>` token in the id; fall back
     to the mzML index attribute.
   - Compute `native_id`: NULL when `native_id_template` is set and
     formatting it with `scan_number` reproduces the source id;
     otherwise store the full id string.
   - Extract scan metadata (filter string, injection time, monoisotopic
     m/z, scan window) and precursor metadata (selected ion m/z, charge,
     intensity, isolation window, activation type, collision energy,
     spectrum_ref).
   - Extract `m/z array` and `intensity array` as numpy arrays in source
     order.
   - Validate equal array lengths.
   - Append one row (metadata + the two arrays) to the current `spectra`
     batch.
9. Flush batches every `batch_size` spectra.
   - Build the batch as a pyarrow table with `mz_array` and
     `intensity_array` as `ListArray` columns of the resolved element
     dtype. A `ListArray` is a flat values buffer plus offsets, so
     memory is proportional to the current batch's peak total, not the
     file's peak total.
   - Register the table and run `INSERT INTO spectra SELECT * FROM
     <tmp>`.
   - Release all batch buffers immediately after the insert.
10. After all spectra are inserted:
    - update `spectrum_count`, `peak_count`, source dtype metadata, storage
      dtype metadata, and any warning metadata;
    - if `index_scan_number=True`, create `idx_spectra_scan_number`;
    - run validation queries;
    - run `CHECKPOINT` so forced compression is applied to the stored
      blocks;
    - close cleanly.

Batching guidance:

- `batch_size=5000` spectra is the default starting point.
- Peak batches should be built from per-spectrum numpy arrays and released
  after insertion.
- If memory use becomes high, reduce `batch_size`. The per-batch memory is
  roughly `batch_size * avg_peaks_per_spectrum * element_size * 2`; for the
  canonical fixture with 5,000 spectra × 248 peaks × 4 bytes × 2 arrays
  this is ~10 MB, well inside typical headroom.

Required post-import validation queries:

```sql
-- 1. Spectrum count.
SELECT COUNT(*) FROM spectra;

-- 2. Total peak count derived from the stored list columns, without
--    materializing every peak through UNNEST.
SELECT SUM(len(mz_array)) FROM spectra;

-- 3. Per-spectrum array-length consistency.
SELECT COUNT(*)
FROM spectra
WHERE len(mz_array) != len(intensity_array);

-- 4. scan_number is unique.
SELECT COUNT(*) - COUNT(DISTINCT scan_number) FROM spectra;

-- 5. scan_number range.
SELECT MIN(scan_number), MAX(scan_number), COUNT(*) FROM spectra;

-- 6. Every row has non-null peak arrays (empty lists are allowed).
SELECT COUNT(*)
FROM spectra
WHERE mz_array IS NULL OR intensity_array IS NULL;
```

The converter must fail if any validation query returns a mismatched
result (spectrum count ≠ expected, mismatched array lengths > 0,
duplicate `scan_number` > 0, or NULL peak arrays > 0).

### 7.2 mzDuck to MGF

Public entry point:

```python
MzDuckFile.open(path).to_mgf(output_path)
```

Algorithm:

1. Open `.mzduck` read-only.
2. Validate schema version.
3. Load `run_metadata` (`rt_unit`, `polarity`, `native_id_template`,
   `mz_array_storage_dtype`, `intensity_array_storage_dtype`).
4. Query spectra ordered by `scan_number`. Single-row SELECT reads
   `mz_array` and `intensity_array` as numpy arrays in source order; no
   UNNEST is needed.
5. For each row:
   - Reconstruct the TITLE id: if `native_id` is non-NULL, use it; else
     format `native_id_template` with `scan_number`.
   - Convert `rt` (stored in `run_metadata.rt_unit`) to seconds.
6. Write an MGF block:

```text
BEGIN IONS
TITLE=<reconstructed native id>
PEPMASS=<precursor_mz> <precursor_intensity>
CHARGE=<precursor_charge>+
RTINSECONDS=<rt converted to seconds>
<mz1> <intensity1>
<mz2> <intensity2>
END IONS
```

MGF rules:

- `TITLE` must use the reconstructed native id (template + `scan_number`
  when `native_id` is NULL).
- `PEPMASS` must use `precursor_mz`; append precursor intensity only when
  it is not NULL.
- `CHARGE` must be omitted when charge is NULL. Positive charges use
  `n+`.
- `RTINSECONDS` must be written when RT is available and convertible.
- If `run_metadata.rt_unit = minute`, multiply by 60.
- If `run_metadata.rt_unit = second`, write the value directly.
- If RT unit is unrecognized, fail with a clear error unless the user
  supplies a conversion override in a future API.
- Peak lines should use enough significant digits to preserve stored
  values.

Reference validation command:

```bash
/data/p/bin/msconvert 1556259.mzML --mgf -o /data2/pub/proteome/PRIDE/protinsight/2019/07/PXD010154/temp/temp/
```

### 7.3 mzDuck to mzML

Public entry point:

```python
MzDuckFile.open(path).to_mzml(output_path)
```

V1 must use:

```python
from psims.mzml import MzMLWriter
```

Algorithm:

1. Open `.mzduck` read-only.
2. Validate schema version and required tables.
3. Load `run_metadata`: `run_id`, `run_start_time`,
   `instrument_config_ref`, `rt_unit`, `polarity`, `centroided`,
   `ion_injection_time_unit`, `native_id_template`,
   `mz_array_storage_dtype`, `intensity_array_storage_dtype`, plus the
   raw-XML provenance snapshots.
4. Initialize `psims.mzml.MzMLWriter`.
5. Write controlled vocabularies, file description, software list,
   instrument configuration, and data processing as closely as psims
   allows.
6. Start one run using `run_id`, `run_start_time`, and instrument config
   metadata when available.
7. For each spectrum ordered by `scan_number`:
   - read one row from `spectra` including `mz_array` and
     `intensity_array`; the lists come back as numpy arrays in source
     order (no UNNEST);
   - reconstruct the mzML `spectrum id`: use `native_id` if non-NULL,
     else format `run_metadata.native_id_template` with `scan_number`;
   - coerce the returned arrays back to their source dtype using
     `mz_array_storage_dtype` / `intensity_array_storage_dtype` so the
     emitted `MS:1000521` / `MS:1000523` CV terms match the original
     mzML;
   - write an MS2 spectrum. `centroided = true` in v1 means emit the
     `centroid spectrum` CV term;
   - include retention time, applying `run_metadata.rt_unit` to the
     `FLOAT` value on the way out;
   - include polarity from `run_metadata.polarity`;
   - include base peak, TIC, and observed m/z bounds when available
     (these are per-row);
   - include scan window when available;
   - include precursor selected ion (from `precursor_mz`),
     `precursor_charge`, `precursor_intensity`, isolation window,
     activation type, collision energy, and `spectrum_ref` when
     available;
   - include filter string, ion injection time (with
     `run_metadata.ion_injection_time_unit`), and monoisotopic m/z when
     psims supports the relevant CV params.
8. Close the writer cleanly so mzML indexes and closing tags are valid.

Round-trip expectation:

- The output mzML must be readable by pyteomics.
- The output mzML should be readable by ProteoWizard where possible.
- XML bytes, attribute ordering, base64 byte layout, and exact header formatting
  are not expected to match the source.
- Supported semantic fields listed in the validation section must match within
  numeric tolerance.

## 9. SQL query examples

Get one spectrum (direct list read, fastest):

```sql
SELECT mz_array, intensity_array
FROM spectra
WHERE scan_number = 42;
```

Get one spectrum via the `peaks` compatibility view:

```sql
SELECT mz, intensity
FROM peaks
WHERE scan_number = 42
ORDER BY peak_index;
```

Find all MS2 spectra for a precursor mass within 10 ppm:

```sql
SELECT scan_number, rt, precursor_mz, precursor_charge
FROM spectra
WHERE ABS(precursor_mz - 500.25) / 500.25 * 1e6 < 10
ORDER BY rt;
```

Extract a product-ion chromatogram. Pruning on `lowest_mz` /
`highest_mz` lets DuckDB skip spectra that cannot contain a peak in the
range before any UNNEST happens:

```sql
SELECT rt, SUM(t.intensity) AS xic
FROM spectra,
     UNNEST(mz_array, intensity_array)
       WITH ORDINALITY AS t(mz, intensity, pos)
WHERE lowest_mz <= 175.120
  AND highest_mz >= 175.118
  AND t.mz BETWEEN 175.118 AND 175.120
GROUP BY rt
ORDER BY rt;
```

Find precursors containing a fragment:

```sql
SELECT DISTINCT scan_number, precursor_mz, precursor_charge, rt
FROM spectra,
     UNNEST(mz_array) AS t(mz)
WHERE lowest_mz <= 175.120
  AND highest_mz >= 175.118
  AND t.mz BETWEEN 175.118 AND 175.120
ORDER BY rt;
```

Charge distribution:

```sql
SELECT precursor_charge, COUNT(*) AS spectra
FROM spectra
GROUP BY precursor_charge
ORDER BY precursor_charge;
```

Activation distribution:

```sql
SELECT activation_type, COUNT(*) AS spectra
FROM spectra
GROUP BY activation_type
ORDER BY spectra DESC;
```

Retention-time window:

```sql
SELECT scan_number, rt, precursor_mz, precursor_charge
FROM spectra
WHERE rt BETWEEN 10.0 AND 20.0
ORDER BY rt;
```

## 10. Failure behavior

### 10.1 Hard errors

The converter must fail with a clear exception or CLI error for:

- missing input file;
- unreadable input file;
- existing output path without `overwrite=True` or `--overwrite`;
- unsupported file extension when the user requests mzML import;
- missing required `scan start time`;
- missing `m/z array` or `intensity array`;
- mismatched m/z and intensity array lengths;
- unsupported MS level in v1 when no explicit filtering option is enabled;
- mixed `rt_unit` across spectra in the same run;
- mixed `polarity` across spectra in the same run;
- mixed `centroided` flag across spectra, or any spectrum with
  `centroided = false` in v1;
- mixed `ion_injection_time_unit` across spectra in the same run;
- `scan_number` cannot be resolved for a spectrum (neither `scan=<N>`
  token nor usable index attribute);
- duplicate `scan_number` after resolution;
- unrecognized schema version when reading;
- failed post-import validation;
- output mzML requested when `psims` is not installed;
- invalid `--compression` value (outside `{zstd, auto, uncompressed}`);
- negative or non-numeric `--compression-level`.

### 10.2 Nullable values

The importer must store NULL for absent optional values:

- precursor m/z;
- precursor charge;
- precursor intensity;
- collision energy;
- activation type;
- isolation window values;
- spectrum_ref;
- base peak values;
- TIC;
- observed m/z bounds when they cannot be computed;
- filter string;
- injection time;
- monoisotopic m/z;
- scan windows;
- `native_id` (NULL when `native_id_template` reproduces the source
  id).

### 10.3 Warnings

The importer should record warnings in `run_metadata` as JSON under
`conversion_warnings` for:

- multiple precursor entries where only the first was stored in core columns;
- unknown activation terms;
- unknown RT unit that was stored but cannot be exported to MGF;
- optional header XML sections that could not be extracted;
- optional metadata fields missing from many spectra;
- `--compression-level` was supplied but the running DuckDB version does
  not accept a numeric zstd level (recorded so users understand why the
  knob had no effect).

Warnings must not hide hard data-integrity errors.

## 14. Design defaults and assumptions

- This document is the source of truth for v1 until replaced by a newer design.
- The documentation update is the only change in the current planning step; it
  does not create package code or tests.
- `mzDuck/` is its own Git repository. Run Git status and diffs from
  `/data/p/xiaolong/mzPeak/mzDuck`.
- V1 is Python-first.
- V1 requires `psims` for mzML export.
- V1 stores one mzML run per `.mzduck`.
- V1 imports centroid MS2 spectra only.
- V1 stores peak arrays as per-spectrum list columns on the `spectra`
  row, not as one row per peak and not in a separate `spectrum_arrays`
  table.
- V1 stores m/z and intensity list element types matching the source
  numpy dtype (FLOAT when source is float32, DOUBLE otherwise) so the
  mzML exporter reproduces the original binary array layout.
- V1 forces DuckDB column compression to `zstd` by default via
  `--compression`; the `--compression-level` CLI knob is exposed and
  recorded even when the running DuckDB version has not yet wired it
  up.
- V1 creates zero secondary indexes by default to match mzMLb's
  indexless layout. `--index-scan-number` is the single opt-in index.
- V1 identifies rows by mzML-native `scan_number`; no mzDuck-invented
  surrogate key exists.
- V1 hoists `rt_unit`, `polarity`, `centroided`,
  `ion_injection_time_unit`, and `native_id_template` into
  `run_metadata` so they are stored once per run.
- mzML round-trip means semantic equivalence for supported fields plus
  byte-equivalent peak arrays when the source dtype is float32, not
  byte-identical XML.
- Retention time is stored in the source unit and converted only when a
  target format requires a specific unit.
- The canonical test file remains the primary integration fixture, and the
  `.mzduck` file size for it must stay close to `1556259.mzMLb`.
