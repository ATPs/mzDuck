# mzDuck V2 Design And Implementation Spec

Date: 2026-04-16  
Updated: 2026-04-17

## Status

This document now describes the intended v2 architecture.
The older v1 notes are obsolete for new writes.

## Goals

V2 is optimized for these constraints:

1. store one mzML run in one portable DuckDB file;
2. keep the output close to mzMLb size in practice;
3. preserve the important mzML semantics needed for round-trip export;
4. avoid storing reconstructable text redundantly;
5. make SQL access straightforward through canonical tables and compatibility
   views;
6. keep the Python package importable through the existing top-level API.

## Schema Versioning

- New writes default to `schema_version = 2`.
- Readers support both v1 and v2.
- v2 is canonical for current development.

## Physical Tables And Views

### Required v2 relations

- `run_metadata`
- `ms2_spectra` when MS2 is included
- `mgf` compatibility view when `ms2_spectra` is present
- `spectrum_text_overrides`
- `spectrum_extra_params`

### Optional v2 relations

- `ms1_spectra`
- `ms3_spectra`, `ms4_spectra`, and higher

### Removed v1 storage

- no physical `spectrum_summary`
- no physical `mgf` table
- no stored `title` column in canonical MS2 storage

## Canonical Storage

### `run_metadata`

`run_metadata(key, value)` stores:

- schema version and provenance;
- source file info and optional SHA-256;
- original mzML header fragments;
- run id and selected run-level attributes;
- counts and peak counts by MS level;
- inferred templates;
- filter-string encoder choice;
- dtype and compression choices;
- table registry and summary values.

### `ms1_spectra`

MS1 rows keep:

- scan identity and source order;
- optional scan-level `instrument_configuration_ref`;
- optional `native_id`;
- typed scan summary fields;
- scan window fields;
- arrays.

### `ms2_spectra`

This is the v2 storage anchor.

It stores:

- scan identity and source order;
- optional scan-level `instrument_configuration_ref`;
- `ms_level`;
- retention time;
- precursor typed fields;
- `precursor_scan_number` instead of stored `spectrum_ref`;
- typed summary fields;
- scan window fields;
- peak arrays.

It does not physically store:

- `title`
- `spectrum_ref`
- `filter_string` when reconstructable
- `native_id` when reconstructable

### `mgf` view

The compatibility view exposes the public MGF contract:

- `scan_number`
- `title`
- `rt`
- `precursor_mz`
- `precursor_intensity`
- `precursor_charge`
- `mz_array`
- `intensity_array`

The title is computed from metadata plus scan number and charge.

### `spectrum_text_overrides`

Sparse exact fallback rows:

```sql
spectrum_text_overrides(
    scan_number INTEGER,
    field_name  VARCHAR,
    value       VARCHAR
)
```

Supported fields:

- `native_id`
- `spectrum_ref`
- `filter_string`

### `spectrum_extra_params`

Unmapped mzML params:

```sql
spectrum_extra_params(
    scan_number     INTEGER,
    scope           VARCHAR,
    ordinal         INTEGER,
    accession       VARCHAR,
    name            VARCHAR,
    value           VARCHAR,
    unit_accession  VARCHAR,
    unit_name       VARCHAR,
    cv_ref          VARCHAR
)
```

This table is for true extra params, not for fields that can be represented as
typed columns or exact templates.

## Reconstruction Rules

### `native_id`

If one exact run-level template exists, store it in `run_metadata` as
`native_id_template`. Otherwise, store only the mismatching rows in
`spectrum_text_overrides`.

### `spectrum_ref`

If one exact template exists, store it in `run_metadata` as
`spectrum_ref_template`, typically:

```text
controllerType=0 controllerNumber=1 scan={precursor_scan_number}
```

Otherwise, store exact raw values in `spectrum_text_overrides`.

### `filter_string`

Filter-string reconstruction is controlled by a registry of candidate encoders.

Rules:

1. detection is per selected run, not global;
2. a candidate is accepted only on exact full-run reconstruction;
3. the accepted id is stored in `run_metadata`;
4. if nothing matches exactly, use `filter_string_encoding = raw`;
5. raw fallback stores original strings rather than approximate guesses;
6. sparse overrides remain possible.

Current registry starts with `thermo_ms2_v1`.

### Scan-level structural attributes

`instrumentConfigurationRef` is stored as a typed per-spectrum field instead of
as a generic extra param row. This reduces file size and exports more cleanly
back to mzML.

## Import Pipeline

1. validate requested import mode;
2. pre-scan the mzML file to gather counts, dtypes, units, and templates;
3. create the v2 schema in a staging DuckDB file;
4. stream spectra into canonical tables in source order;
5. collect sparse text overrides and remaining extra params;
6. run exact filter-string encoder detection over the selected spectra;
7. delete raw `filter_string` overrides if an exact encoder matched;
8. compute summary metadata and table registry;
9. checkpoint the staging file;
10. compact into a fresh final database with `COPY FROM DATABASE`;
11. validate the compacted output;
12. atomically rename the compacted file into place.

## Export Behavior

### `MzDuckFile.get_spectrum()`

The public API reconstructs:

- `title`
- `native_id`
- `spectrum_ref`
- `filter_string`

It also promotes structural scan fields such as
`instrument_configuration_ref`.

### MGF export

`export_mgf()` always derives `TITLE` from metadata and never reads a stored
title column.

### mzML export

`export_mzml()`:

- reads spectra in source order when `source_index` is available;
- reconstructs typed text fields from templates plus sparse overrides;
- merges `spectrum_extra_params`;
- restores stored mzML header fragments;
- restores original `defaultDataProcessingRef` on `spectrumList`;
- restores per-scan `instrumentConfigurationRef`.

## Compaction

DuckDB `VACUUM` was not enough on the PRIDE file.
The required v2 write path is:

```text
staging import -> CHECKPOINT -> COPY FROM DATABASE -> validate -> rename
```

This compact copy step is mandatory for normal output.

## CLI Requirements

### `mzduck --help`

The top-level help page must act as the landing page and explain:

- the file model;
- canonical tables and compatibility views;
- what is reconstructed vs physically stored;
- filter-string encoder detection and raw fallback;
- compaction behavior;
- round-trip scope;
- end-to-end examples.

### `mzduck convert --help`

Keep the convert help focused and readable.
It should keep the current command structure, but describe v2 terms:

- canonical `ms2_spectra`
- derived `mgf`
- no physical `spectrum_summary`
- `--ms2-mgf-only` and `--ms2-only` sharing the same compact storage shape

## Validation Expectations

### Fast regression tests

- v2 schema validation
- `mgf` compatibility view shape
- no physical `spectrum_summary`
- stable top-level imports
- `get_spectrum()`, `export_mgf()`, `export_mzml()`
- detailed `mzduck --help`
- focused `mzduck convert --help`

### Focused unit tests

- spectrum-ref template detection
- filter-string encoder detection
- raw fallback when no exact encoder exists
- sparse override behavior
- mzML header restoration
- scan-level instrument configuration export

### Large-run validation

When the PRIDE files are available, validate:

- exact recovery of all filter strings;
- exact recovery of all precursor spectrum references;
- exact recovery of scan order;
- exact recovery of scan-level instrument configuration references;
- compacted output size near the target range;
- successful export back to mzML with preserved run header metadata.
